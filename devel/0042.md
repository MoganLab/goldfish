## 2026/05/27 性能优化

### What
利用 `(liii timeit)` 定位热点并优化 `(liii json)` 核心函数性能。

1. **重写 `json->string` 序列化**：将 `goldfish/guenchi/json.scm` 中递归 `string-append` 的实现改为基于 `open-output-string` 输出端口，消除 O(n²) 的字符串拼接开销。
2. **简化 `ensure-json-structure`**：将全量 `every pair?` 遍历改为轻量的顶层形状检查 `(or (and (list? x) (not (null? x))) (vector? x))`，避免每次 `json-ref`/`json-set` 等操作都遍历整个对象。
3. **减少重复类型检查**：`json-ref`、`json-set`、`json-push`、`json-drop`、`json-reduce` 中对空对象 `'(())` 的判断，去掉冗余的 `json-object?` 调用。
4. **优化 `json-contains-key?`**：改为 `(and (pair? json) (not (equal? json '(()))) (assoc key json))`，避免 `every pair?` 遍历。
5. **优化 `json-keys`**：改为直接 `(if (and (list? json) (not (null? json)) (not (equal? json '(())))) (map car json) '())`，去掉 `every pair?` 遍历。
6. **新增 `bench/json-perf.scm`**：使用 `(liii timeit)` 对解析、序列化、读取、键检查等操作做 micro-benchmark。

### Why
`(liii json)` 是纯 Scheme 实现，大对象上的线性遍历和字符串拼接导致明显的性能瓶颈。

### How
- 序列化瓶颈：原 `json->string` 递归拼接中间字符串，对象越大复制次数越多。改为向输出端口 `display`，只需一次内存分配。
- 访问瓶颈：`ensure-json-structure` 调用 `json-object?`，后者用 `every pair?` 遍历整个 alist。改为只检查是否为非空 list 或 vector，将逐元素验证延迟到实际操作时自然暴露。

### 已知行为变化
- `json-keys` 对非法 JSON 对象（如包含非 pair 元素的列表）不再静默返回 `'()`，而可能抛出 `wrong-type-arg` 异常。已在 `tests/liii/json/json-keys-test.scm` 文档中说明。
- `json-contains-key?` 对非法 JSON 对象不再保证返回 `#f`，`assoc` 可能在遍历到非法元素前命中匹配键。

### 性能对比（200 键对象 / 200 长度数组）

| 操作 | 优化前 | 优化后 | 提升 |
|---|---|---|---|
| 序列化对象 `json->string` | 0.065s | 0.048s | **26%** |
| 序列化数组 `json->string` | 0.0032s | 0.0012s | **62%** |
| 对象读取 `json-ref` | 0.063s | 0.012s | **81%** |
| 键存在 `json-contains-key?` | 0.025s | 0.00027s | **99%** |
| 获取键 `json-keys` | 0.0069s | 0.00082s | **88%** |
