## 2026-07-19 用 C++ 重写 5 张超长字符表

### What
1. 性能归因：`(import (liii json))` 慢 → `(import (liii list))` 慢（~135ms）→ `(scheme char)` 慢（~130ms），
   原因是 char.scm 加载时用 Scheme 循环执行数千次 `s7-hash-table-set!` 构建 5 张码点表。
2. 新建 src/scheme_char.cpp，将 5 张特别长的表移植为 C++ 静态有序区间表 + 二分查找：
   - `char-upcase`（1477 项映射）
   - `char-downcase`（1460 项映射）
   - `char-alphabetic?`（非字母反向表，757 个区间 / 62985 码点）
   - `char-upper-case?`（1858 项）
   - `char-lower-case?`（2569 项）
3. 小表（foldcase/numeric/whitespace/digit-value）保留 Scheme 实现。
4. char.scm 相应函数改为薄封装，调用 g_* C 函数。

### Why
- 解决 `(import (liii json))` / `(import (liii list))` 的加载性能瓶颈（(scheme char) 占 ~130ms）。
- C++ 静态表零初始化成本，二分查找同时提升函数本身的运行性能。

### How
1. TDD：先用 tools/parse_char_tables.py 解析 char.scm 的 5 张表，为覆盖不足的函数
   （upcase/downcase/lower-case?/alphabetic?）生成穷尽或区间边界测试，确认基线全绿。
2. 同一解析器合并连续区间（映射表按 delta 相同合并）生成 C++ 静态数组。
3. g_* 函数在 C 层完成类型检查（type-error）与查表，语义与 Scheme 版逐项一致：
   - alphabetic?: cp > 205743 → #f；命中非字母区间 → #f；否则 #t
   - upcase/downcase: 未命中返回原字符

### 结果（低负载下测量）
- import 性能：
  - `(scheme char)` 冷加载 ~130ms → ~6ms（约 21 倍）
  - `(liii list)` 冷加载 ~135ms → ~11ms（约 12 倍）
  - `(liii json)` 冷加载 ~16ms → ~7ms
- 函数调用性能（10 万次平均，bench/char-perf.scm）：
  - `char-upcase` 7.9µs → 0.46µs（约 17 倍）
  - `char-downcase` 12.2µs → 0.62µs（约 20 倍）
  - `char-alphabetic?` 8.6µs → 0.63µs（约 14 倍）
  - `char-upper-case?` 3.6µs → 0.66µs（约 5.5 倍）
  - `char-lower-case?` 4.2µs → 0.66µs（约 6.4 倍）
- 测试：`bin/gf test --all` 1451 通过 / 21 失败，21 个失败在原始 main 上同样存在
  （如 tests/liii/list/take-test.scm 在未修改的 main 上也 SIGABRT），与本改动无关。
