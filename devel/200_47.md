## 2026/03/31 移除 rich-path，重新实现 path 模块

### What
移除 `(liii rich-path)` 模块，将其功能完全整合到 `(liii path)` 中，同时清理相关依赖。

1. 完全重写 `(liii path)` 模块：
   - 使用 record 类型定义 path 对象
   - 实现完整的 path API：`path`, `path?`, `path-copy`, `path-parts`, `path-type`, `path-drive`
   - 实现路径操作：`path-name`, `path-stem`, `path-suffix`, `path-absolute?`, `path-relative?`, `path-parent`, `path-join`
   - 实现文件操作：`path-dir?`, `path-file?`, `path-exists?`, `path-read-text`, `path-write-text`, `path-read-bytes`, `path-append-text`, `path-touch`, `path-list`, `path-rmdir`, `path-unlink`

2. 更新 `src/goldfish_repl.cpp`：
   - 移除对已删除文件（`lang.scm`, `rich-path.scm`, `rich-list.scm` 等）的加载

3. 清理测试文件：
   - 从 82+ 个测试文件中移除 `(liii lang)` 导入
   - 修改 `listdir-test.scm` 使用 `vector-contains?` 代替 `$` 操作符
   - 修改 `http-test.scm` 使用标准 `display` 代替 `display*`
   - 删除 `datetime-test.scm`（依赖已删除的模块）

4. 更新 `goldfish/srfi/srfi-78.scm`：
   - 使用 `equal?` 代替 `class=?`
   - 使用 `display` 代替 `display*`

5. 清理 bench 和 demo 文件：
   - 删除依赖已删除模块的 bench 和 demo 文件

### Why
`(liii rich-path)` 提供了基于 case-class 的面向对象路径操作，但它：
- 依赖 `(liii lang)` 和 `(liii oop)` 等重量级模块
- 使用宏实现，有显著的性能开销
- 增加了代码复杂度

将其功能整合到纯函数的 `(liii path)` 中可以：
- 简化代码结构
- 减少模块依赖
- 提高运行时性能
- 保持向后兼容的 API

### How
1. 使用 Scheme 的 `define-record-type` 定义 path 记录类型
2. 实现 `path->string` 作为核心转换函数，处理 POSIX 和 Windows 路径格式
3. 实现 `parts->string` 辅助函数，正确处理绝对路径和相对路径
4. 重写 `path-name`, `path-stem`, `path-suffix` 等函数，处理特殊边界情况
5. 批量移除测试文件中未使用的 `(liii lang)` 导入
