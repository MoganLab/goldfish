## 2026-05-04 (scheme base) 数值函数内置名加 s7- 前缀

### What
将 `goldfish/scheme/base.scm` 中通过 `(define s7-xxx xxx)` 做别名适配的数值函数，直接在 `s7.c` 中内置为带 `s7-` 前缀的名称。

涉及的函数列表（共 8 个）：

| s7.c 当前内置名 | scheme/base.scm 中别名 | 方案 |
|---|---|---|
| `floor` | `s7-floor` | s7.c 改为 `s7-floor`，base.scm 删除别名 |
| `ceiling` | `s7-ceiling` | s7.c 改为 `s7-ceiling`，base.scm 删除别名 |
| `truncate` | `s7-truncate` | s7.c 改为 `s7-truncate`，base.scm 删除别名 |
| `round` | `s7-round` | s7.c 改为 `s7-round`，base.scm 删除别名 |
| `max` | `s7-max` | s7.c 改为 `s7-max`，base.scm 删除别名 |
| `min` | `s7-min` | s7.c 改为 `s7-min`，base.scm 删除别名 |
| `modulo` | `s7-modulo` | s7.c 改为 `s7-modulo`，base.scm 删除别名 |
| `lcm` | `s7-lcm` | s7.c 改为 `s7-lcm`，base.scm 删除别名 |

具体修改步骤（以 `floor` 为例，其余同理）：

1. **修改 `src/s7.c`**：
   - 将 `defun("floor", ...)` 改为 `defun("s7-floor", ...)`
   - 更新 `sc->floor_symbol` 等绑定（变量名可保留，仅改字符串）
   - 更新帮助文本中的函数名

2. **简化 `goldfish/scheme/base.scm`**：
   - 删除 `(define s7-floor floor)` 别名
   - `floor` 函数体中直接调用 `s7-floor`（此时 `s7-floor` 是 s7.c 内置名）
   - **export 列表调整**：不再导出 `s7-floor` 等内部别名

3. **新增/更新测试文件**：
   - 在 `tests/goldfish/scheme/base/` 下为上述函数补充测试
   - 确保 `bin/gf tests/goldfish/scheme/base-test.scm` 回归通过

### Why
`(scheme base)` 通过 `export` 暴露了 `floor` 等函数，但这些函数在 `begin` 块中被重定义（例如加入了 inexact 转换、类型检查等）。为了访问原始的内置版本，代码中不得不先 `(define s7-floor floor)` 保存别名，再重新定义 `floor`。这导致 `(import (scheme base))` 的语义不一致：

- 如果先 `import` 再使用，得到的是重定义后的版本；
- 如果其他库在 `(scheme base)` import 之前就引用了 `floor`，可能拿到的是 s7 内置原版。

直接在 s7.c 中将内置名加上 `s7-` 前缀，可以彻底消除这层别名适配，使得 `scheme/base.scm` 中的定义成为真正的、无歧义的 R7RS 实现。

### How
- **仅修改 s7.c 中 `defun` 注册的 Scheme 层面函数名**。C API 函数（如 `s7_floor`）、内部 C 变量名（如 `sc->floor_symbol`）保持不变。
- 由于 `floor`/`ceiling` 等函数是通过 `s7_define_typed_function` 或 `defun` 注册的，修改时只需调整第一个参数（Scheme 名称字符串）。
- `floor-quotient`、`floor-remainder`、`floor/`、`truncate/` 等由 `scheme/base.scm` 自行实现的函数不受此改动影响。
- 注意检查 `set_is_translucent` 等附加属性是否随符号绑定正确迁移。

## 2026-05-04 重命名 make-hash-table 和 hash-table-set!

### What
将 s7.c 中内置的 `make-hash-table` 和 `hash-table-set!` 函数分别改名为 `s7-make-hash-table` 和 `s7-hash-table-set!`。

1. 修改 `src/s7.c`：
   - 将 `defun("make-hash-table", ...)` 改名为 `defun("s7-make-hash-table", ...)`
   - 将 `defun("hash-table-set!", ...)` 改名为 `defun("s7-hash-table-set!", ...)`
   - 更新帮助文本字符串中的函数名

2. 简化 `goldfish/srfi/srfi-125.scm`：
   - 删除 `(define s7-hash-table-set! hash-table-set!)`
   - 删除 `(define s7-make-hash-table make-hash-table)`
   - 直接调用内置的 `s7-hash-table-set!` 和 `s7-make-hash-table`

### Why
`srfi-125.scm` 导出了 `make-hash-table` 和 `hash-table-set!`，但 s7.c 也内置了同名函数。为了避免冲突，`srfi-125.scm` 不得不在 `begin` 块开头做别名适配：先把内置函数保存为带 `s7-` 前缀的别名，再重新定义同名函数。这导致 `(import (liii hash-table))` 的 import 行为不一致——有时导入的是 s7 内置版本，有时导入的是 srfi-125 重定义版本。

将 s7 内置函数直接加上 `s7-` 前缀后，`srfi-125.scm` 中不再需要别名适配，import 行为变得一致且明确。

### How
仅修改 s7.c 中 `defun` 注册的 Scheme 层面函数名。C API 函数 `s7_make_hash_table` 和 `s7_hash_table_set` 的 C 层面名称保持不变，内部使用的 `make_hash_table_symbol` 和 `hash_table_set_symbol` 等 C 变量名也不变。

## 2026-05-04 之前的提交和任务描述

### What
将 s7.c 中内置的 `hash-table-entries` 函数改名为 `hash-table-size`。

1. 修改 `src/s7.c`：
   - 将 `g_hash_table_entries` 改名为 `g_hash_table_size`
   - 将 `hash_table_entries_i_7p` 改名为 `hash_table_size_i_7p`
   - 将 `sc->hash_table_entries_symbol` 改名为 `sc->hash_table_size_symbol`
   - 更新 `defun` 注册和 `s7_set_i_7p_function` 绑定

2. 简化 `goldfish/srfi/srfi-125.scm`：
   - 删除 `(define s7-hash-table-entries hash-table-entries)`
   - 删除 `(define hash-table-size s7-hash-table-entries)`
   - 内置的 `hash-table-size` 可直接使用

### Why
s7.c 内置的 `hash-table-entries` 语义是"返回哈希表中的条目数量"，但函数名容易让人误解为"返回条目本身"。

SRFI-125 标准中：
- `hash-table-size` 返回条目数量
- `hash-table-entries` 返回两个值（keys 和 values）

当前 s7.c 的命名与 SRFI-125 冲突，导致 `srfi-125.scm` 不得不做绕弯的重命名适配：先把内置 `hash-table-entries` 保存为 `s7-hash-table-entries`，再用它实现 `hash-table-size`，最后重新定义 `hash-table-entries`。改名后可以消除这层冗余。

### How
直接在 s7.c 中全局替换符号名和函数名。`hash_table_entries(p)` 宏（定义在 `s7.c` 内部，表示哈希表结构体的条目计数字段）保持不变，仅暴露给 Scheme 层面的函数名从 `hash-table-entries` 改为 `hash-table-size`。
