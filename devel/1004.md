## 2026/06/16 修复 Windows 上 `path-parent` 对正斜杠路径解析错误

### What
1. `parse-windows-path` 原本只在 `(os-sep)`(`\`)上做 `string-split-vec`,Windows API 接受但 scheme 层不识别 `/`,导致带正斜杠的绝对路径(如 `C:/Users/foo/bar.scm`)整串被当作单一 part
2. `path-parent` 在 string 输入分支同样只用 `(os-sep)`,即便外层把字符串包装成 path object 之前先经过 `path->string` 直通(返回原串),也只识别单种分隔符
3. 两处统一参照 `parse-path-string` 已有的写法,在 Windows 上先把 `s` 中的 `/` 替换为 `\`,得到 `normalized`,后续判断、split、loop 都基于 `normalized`。POSIX 平台不做替换,保持原有行为

### Why
- s7 内置的 `(port-filename)` 在脚本被 `s7_load` 时返回脚本路径,在 Windows 上是带正斜杠的绝对路径(如 `C:/Users/.../foo.scm`),与 `(command-line)` 第二项风格一致
- goldfish 的下游(Mogan 的 autosave 插件)入口脚本顶部用 `(path->string (path-parent (port-filename)))` 把脚本所在目录加入 `*load-path*`。原 bug 下 path-parent 把整串当单一 part,剥到只剩 `C:`,扩展出来的 `*load-path*` 条目是错误的目录,后续 `(import (liii autosave))` 找不到模块,插件进程握手前即崩溃,表现为「Windows 上 autosave 不工作」
- Linux/macOS 上之所以一直没暴露,是侥幸——POSIX 路径里 `/` 就是 sep,`string-split-vec` 一切正常。`parse-path-string` 在非 drive-letter 路径上已经做了相同的归一化(line 87 附近),`parse-windows-path` 漏了,`path-parent` 也漏了。补齐两处,Windows 与 POSIX 行为对齐
- 选择修 goldfish 上游而不是让下游每个插件入口各写一份 `argv`/`getcwd` fallback:后者是绕过 root cause 的局部 workaround,每加一个插件入口都得重写一遍,且掩盖 goldfish 的 bug 让其它用 path API 的代码也踩同样的坑

### How
- `parse-windows-path`:加一层 `(let ((normalized (string-replace s "/" "\\"))) ...)` shadow 外层,后续绝对路径判定(`(char=? (string-ref normalized 2) #\\)`)与 `substring`/`string-split-vec` 都基于 `normalized`
- `path-parent`:在 `(path->string p)` 之后加一层 `(let ((s (if (os-windows?) (string-replace s "/" "\\") s))) ...)` shadow 外层 `s`,后续 `s-trimmed` 与 loop 都基于规范化后的 s。`string-replace` 用 `(liii string)` 的签名 `(string-replace str old new)`,同文件 `parse-path-string` 已经在用

### 验证(Mogan 端)
- `goldfish .../tm-autosave.scm` 输出 `verbatim:autosave`,不再报 `No such file or directory "liii/autosave.scm"`
- 临时打印 `(path->string (path-parent (port-filename)))`,对脚本绝对路径 `C:/Users/.../tm-autosave.scm` 正确得到 `C:\Users\...\goldfish\`
- 端到端在 Mogan 中触发保存,`$TEXMACS_HOME_PATH/system/backup/<doc-id>/` 下生成带时间戳的 `.tmu`

### Notes
- 修改同时同步到 Mogan 内嵌的 `TeXmacs/plugins/goldfish/goldfish/liii/path.scm`(mogan 不引用 goldfish 仓库,运行时加载自己那一份 .scm)
- 不改 `(string-split-vec)` 本身去支持「分隔符集合」——单字符 sep 接口简单,所有调用点都依赖这一假设;在调用前归一化是更小的改动
- 不修 `path->string` 让 string 输入自动归一化——会改变 string 直通的语义、影响范围不可控
