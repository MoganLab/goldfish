### 2026/06/23 Windows normalize 0 差异 + path.scm 组织重构
**验收**:`python3 pathlib-ref/normalize.py windows` 119 项 **0 差异**(原 26 项),Windows 平台 `(liii path)` 与 `PureWindowsPath` 完全一致。

**功能性修复(4 处):**
1. `path-drive` 加冒号:Windows 单字符盘符 `C` → `C:`(对齐 `PurePath.drive`,原存储字段无冒号)。UNC 的 `\\server\share` 整体保持无冒号。
2. 删除旧的 `path-root` 静态构造器(无 pathlib 对应、无使用场景,纯属 goldfish 历史 API 债务);新增同名的 `path-root` **字符串访问器**(对齐 `PurePath.root`):有根 posix 返回 `"/"`、Windows 返回 `"\"`;无根返回 `""`(包括仅 server 无 share 的 `\\srv`)。原静态构造器调用点改为内联 `make-path-record`,涉及测试 `path-root`/`path-of-drive`/`path-absolute-p`/`path-join`/`path-getsize`/`path-parent`/`path-type`/`path-to-string`/`path-parts` 全部从 `(path-root)` 改为 `(path "/")`,Windows 平台相关的断言用 `(when (not (os-windows?)) ...)` 包裹。
3. `path-match` Windows 类型大小写不敏感(对齐 `PureWindowsPath.match`,原大小写敏感):`(path-match (path "Foo.TXT") "*.txt")` → `#t`(原 `#f`)。posix 类型保持敏感。
4. `parse-unc` 修 `\\srv`(仅 server,无 share)root 字段 bug:`#\\` → `#f`(对齐 `PureWindowsPath('\\\\srv').root == ''`)。

**代码组织重构(`goldfish/liii/path.scm`):**

- 每个 `path-*` 公共函数加 pathlib 对齐标注(`;; 对齐 pathlib.PurePath.xxx`)+ Windows/posix 差异说明。文件系统操作类函数(`path-exists?` 等)加一行效果注释,不标 pathlib 对应。

**测试改动:**

**新增 `pathlib-ref/` 对照工具**(根目录):把"goldfish 行为是否等于 pathlib"从依赖手写断言,升级为可复现的逐项对照。

- `README.md` — 用法与平台说明(Windows 审计须在 Windows 平台跑)。

**边界审计发现并修复的 POSIX bug**(经 normalize 对照 148 项,0 差异):
1. `path-stem`/`path-suffix`/`path-suffixes` 对末尾点文件 `foo.` 错误切分:pathlib `foo.` 的 stem=`foo.`、suffix=空、suffixes=空(末尾点不构成后缀);原实现给 stem=`foo`、suffix=`.`。修 `split-name-dots` 和 `path-suffixes`:末段为空时不切。
2. `path-join` 空串段错误追加 `.`:pathlib `P('/a').joinpath('')` => `/a`(空串是 no-op)、`P('').joinpath('b')` => `b`(空 base 不引入 `.`);原实现给 `/a/.`、`./b`。修 `path-join`:loop 跳过空串段,`append-parts` 过滤 `.` 段。
3. `path-parts` 对当前目录 `.` 返回 `#(".")`:pathlib `PurePath('.').parts` => `()`;原返回 `#(".")`。修 `path-parts`:纯当前目录返回空向量。

**对照结论**:POSIX 平台 goldfish 与 `PurePosixPath` **148 项 0 差异,完全一致**;Windows 待换平台跑 `normalize.py windows` 确认。

### 2026/06/23 全面对齐 pathlib 的平台相关行为
**核心认知**:pathlib 行为本身平台相关(`Path("C:\\a")` 在 POSIX 是 `PurePosixPath`,在 Windows 是 `PureWindowsPath`)。goldfish 已正确按 `os-windows?` 分流 type(已验证 macOS 上 `(path "C:\\a")` type 为 `'posix`)。本轮把所有测试断言对齐到 pathlib 在对应平台的真值(全部经 `python3` 实跑验证)。

**实现修复(POSIX 本机全绿,Windows 项待 CI):**
1. `path-with-stem` 多后缀只留最后一个 suffix(对齐 `with_stem`):`a.tar.gz`+`new` → `new.gz`(原 `new.tar.gz`)。
2. 相对 `path-parents` 补 `.` 终点:`a/b/c` → `(a/b a .)`、`a` → `(.)`、`.` → `()`(原相对路径不补 `.`)。
3. `path-absolute?` Windows `\foo` 改为非绝对(drive 空时即使有 root 也不绝对):**反转 eeba3da4 的结论**——经 `PureWindowsPath('\\foo').is_absolute()` 实跑确认为 `False`。Windows 绝对 = drive 非空 AND root 非空。
4. UNC anchor 字符串加尾斜杠:`str(W('\\srv\sh'))` == `'\\srv\sh\\'`;`parent`/`parents` 末元素同步。光秃 server `\\srv`(无 share)仍不带尾斜杠。
5. `path-join` 改用 record 累加器(返回 path record):`C:\a`+`\b` → `C:\b`(继承 base drive);`C:\base`+`D:rel` → `D:rel\x.txt`(不同 drive 重置);`C:\base`+`C:rel` → `C:\base\rel`(同 drive 合并)。
6. `path-parts` Windows 分支加 anchor 首元素:`C:\tmp\demo.txt` → `#("C:\\" "tmp" "demo.txt")`;UNC → `#("\\\\srv\\sh\\" "a" "b")`;`path-from-parts` 互逆重建(新增 UNC anchor 识别、区分 `C:`/`C:\`)。
7. `path-name` 改为 record parts 取末段(原用平台 sep,跨 type 场景失效)。
8. `path-match` 扩展支持含分隔符模式:`a/b/c`.match(`b/c`) → True;绝对模式要求段数完全匹配。

**补充缺失测试(POSIX 本机):** with-suffix 无点报 value-error、去多后缀只去最后一个;suffixes 连续点 `a..b`;unlink 目录报 value-error;list 空目录/不存在目录;mkdir parents:#t 叶子已存在报错。

**有意偏离(保留):** `path-rename` 目标已存在报 `file-exists-error`(pathlib/POSIX 是覆盖),goldfish 是更安全的有意设计,测试加注释。

### 2026/06/22 修 path-parent 在 UNC 单段路径下的 bug
**Bug:** `(path-parent (path "\\\\srv\\sh\\a"))` 返回 `\:\`。
**Why:** `(= n 1)` 分支只区分 posix / 单字母盘符 / 无 drive。对 UNC 路径,`drive` 字段存的是整个 share anchor(`"\\\\srv\\sh"`,以 `\\` 开头),原代码走 `(path-of-drive (string-ref drive 0))` 把 `#\\` 当盘符字母处理 → 字符串塌成乱码。
**How:** 在 `path-parent` 的 `(= n 1)` 分支加 `(unc-prefix? drive)` 判定,UNC 单段路径直接回到 `make-path-record #() 'windows drive root`(空 parts + 原 UNC anchor),对齐 `pathlib.PureWindowsPath("\\\\srv\\sh\\a").parent == "\\\\srv\\sh"`。
