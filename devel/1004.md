## 2026/06/16 重写 Windows 路径解析,对齐 pathlib 语义

### What

1. 给 `<path>` record 增加 `root` 字段,用于表达绝对性。此前绝对性靠 parts 向量里的 `""` stub(`posix`)或 drive 字段(`windows`)隐含表达,导致 `C:\foo`(drive-absolute)与 `C:foo`(drive-relative)在 record 内部不可区分,后续 `path->string`、`path-absolute?` 都拿不准语义。
2. 重写 `parse-windows-path` 为独立模块 `parse-unc` + 主分发,覆盖四种 Windows 路径形式:`\\srv\sh\path`(UNC)、`C:\path`(drive-absolute)、`C:foo`(drive-relative)、`\foo`(current-drive root)。对齐 `pathlib.PureWindowsPath` 的 `.drive`/`.root`/`.parts` 分解。
3. 解析阶段过滤纯 `.` 段(`a/./b` → `a/b`),保留 `..`(只在 `resolve()` 才处理)。新增 `drop-dot-parts`,内部用 `(liii vector)` 的 `vector-filter` 实现。
4. `path-absolute?` Windows 分支改为同时要求 drive 与 root 非空,修正 `C:foo` 被误判为绝对的 bug(此前规则只看 drive 非空)。字符串回退分支统一改走 `(path-absolute? (path p))`,消除"传 path 与传 string 语义不一致"的隐患。
5. `path` 入口对 UNC/drive 路径识别加 `(os-windows?)` 守卫。非 Windows 平台一律按 posix 处理,避免 `\\foo` 在 Linux 上被误当 UNC 污染调用方。
6. 大量手写 `let loop` 改用 `(liii string)` / `(liii vector)` 现成函数:
   - `string-split-vec` 用 `(list->vector (string-split ...))` 替代手写字符遍历
   - 删掉手写 `string-index-of`,用 `(string-index ...)` 与 `(string-index-right ...)`
   - `parts->string` 用 `(string-join (vector->list parts) sep)` 替代手写拼接循环,行数从 ~60 行降到 5 行
   - `path-name` / `path-parent` 用 `string-index-right` 找末次分隔符
   - `path-stem` 用 `string-join` 拼回
7. 新增 `parse-unc` 辅助函数,把 UNC 解析从 `parse-windows-path` 内联的多层嵌套 cond 中抽出,缩进与可读性显著改善。

### Why

- 原 `<path>` record 把 drive-absolute 与 drive-relative 编码成同一组字段值(parts/drive 相同,只有起始字符差异),数据模型有歧义。`path->string` 拿不准该输出 `C:\foo` 还是 `C:foo`,导致 `path-absolute?` 在 Windows 上对 `C:foo` 返回 `#t`(语义错)。
- Mogan autosave 插件在 Windows 上的备份路径用 `(path-join cwd "backup")` 拼接,如果 cwd 是 drive-relative 或被误判为绝对,join 结果会出现盘符重复或路径错乱。修了 record 模型才能彻底消除这一类 bug。
- pathlib(Python `pathlib.PureWindowsPath`)是事实上的语义标准,与其让 Scheme 版本走自己的岔路,不如直接对齐——`.drive`/`.root`/`.parts` 的分解逻辑在 Python 标准库里已被验证过,跨平台行为可预期。
- 大量手写循环本可由 `(liii string)` / `(liii vector)` 的现成函数替代,留着是历史包袱。借这次 record 改动顺手清理,后续读 path.scm 不必每次重新理解一堆 `let loop` 的状态机。

### How

- **root 字段语义**:`#\\`(任意 char,真值即可)表示路径有根分隔符,`#f` 表示无根。
  - `path-of-drive` 构造的 `C:\` → drive=`"C"`、root=`#\\`、parts=`#()`(对外契约文档承诺"返回驱动器根",保持不变)
  - `(path "C:\\foo")` → drive=`"C"`、root=`#\\`、parts=`#("foo")`
  - `(path "C:foo")` → drive=`"C"`、root=`#f`、parts=`#("foo")`(语义差异完全落在 root 字段)
- **path-parts 对外契约**:为不破坏历史 caller,getter 在 posix 绝对路径(`type='posix` + `drive=""` + `root=#\/`)上重新加上 `"/"` stub 返回;UNC/drive 路径上不加 stub(因为 drive 字段已表达绝对性)。原 `(path-parts (path-from-parts #("/" "tmp" "demo.txt"))) => #("/" "tmp" "demo.txt")` 用例保持不变。
- **UNC 识别的平台守卫**:UNC `\\` 前缀只在 `(os-windows?)` 为真时识别。Linux/macOS 上 `\\foo` 走 posix 分支当普通路径处理,行为与原版一致。
- **`drop-dot-parts` 不导入 `(liii list)`**:`(liii list)` 同时导出 `remove`(谓词版),会和 `(liii os)` 的文件 `remove` 冲突,导致 path-unlink 调用 `(remove s)` 时被错误解释为谓词版。改用 `(liii vector)` 的 `vector-filter`,零命名冲突。
- **不破坏 `path-of-drive` 文档承诺**:该函数的对外契约文档明确写"返回驱动器根 `C:\`",且 path-absolute-p-test.scm 在 Windows 与非 Windows 平台都断言 `(path-absolute? (path-of-drive #\C))` 为真。重构后在内部填 `root=#\\`,对外 `path->string` 仍输出 `C:\`、`path-absolute?` 仍返回 `#t`。

### Notes

## 2026/06/16 修复 Windows 上 `path-parent` 对正斜杠路径解析错误

### What
1. `parse-windows-path` 原本只在 `(os-sep)`(`\`)上做 `string-split-vec`,Windows API 接受但 scheme 层不识别 `/`,导致带正斜杠的绝对路径(如 `C:/Users/foo/bar.scm`)整串被当作单一 part
2. `path-parent` 在 string 输入分支同样只用 `(os-sep)`,即便外层把字符串包装成 path object 之前先经过 `path->string` 直通(返回原串),也只识别单种分隔符
3. 两处统一参照 `parse-path-string` 已有的写法,在 Windows 上先把 `s` 中的 `/` 替换为 `\`,得到 `normalized`,后续判断、split、loop 都基于 `normalized`。POSIX 平台不做替换,保持原有行为

### Why
- s7 内置的 `(port-filename)` 在脚本被 `s7_load` 时返回脚本路径,在 Windows 上是带正斜杠的绝对路径(如 `C:/Users/.../foo.scm`),与 `(command-line)` 第二项风格一致
- goldfish 的下游(Mogan 的 autosave 插件)入口脚本顶部用 `(path->string (path-parent (port-filename)))` 把脚本所在目录加入 `*load-path*`。原 bug 下 path-parent 把整串当单一 part,剥到只剩 `C:`,扩展出来的 `*load-path*` 条目是错误的目录,后续 `(import (liii autosave))` 找不到模块,插件进程握手前即崩溃,表现为「Windows 上 autosave 不工作」
- Linux/macOS 上之所以一直没暴露,是侥幸——POSIX 路径里 `/` 就是 sep,`string-split-vec` 一切正常。`parse-path-string` 在非 drive-letter 路径上已经做了相同的归一化(line 87 附近),`parse-windows-path` 漏了,`path-parent` 也漏了。补齐两处,Windows 与 POSIX 行为对齐
- 选择修 goldfish 上游而不是让下游每个插件入口各写一份 `argv`/`getcwd` fallback:后者是绕过 root cause 的局部 workaround,每加一个插件入口都得重写一遍,且掩盖 goldfish 的 bug 让其它用 path API 的代码也踩同样的坑

### How
- `parse-windows-path`:加一层 `(let ((normalized (string-replace s "/" "\\"))) ...)` shadow 外层,后续绝对路径判定(`(char=? (string-ref normalized 2) #\\)`)与 `substring`/`string-split-vec` 都基于 `normalized`
- `path-parent`:在 `(path->string p)` 之后加一层 `(let ((s (if (os-windows?) (string-replace s "/" "\\") s))) ...)` shadow 外层 `s`,后续 `s-trimmed` 与 loop 都基于规范化后的 s。`string-replace` 用 `(liii string)` 的签名 `(string-replace str old new)`,同文件 `parse-path-string` 已经在用
