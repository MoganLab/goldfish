### 文件命名规则
本次迁移沿用统一的特殊符号转写规则：

1. `?` -> `-p`
2. `!` -> `-bang`
3. `->` -> `-to-`
4. `=` -> `-eq`
5. `*` -> `-star`
6. `/` -> `-slash-`
7. 关系运算符中的 `<` / `>` / `<=` / `>=` 按语义写法转写

本次具体文件名示例：
- `set-contains?` -> `set-contains-p-test.scm`
- `set-empty?` -> `set-empty-p-test.scm`
- `set=?` -> `set-eq-p-test.scm`
- `set<=?` -> `set-le-p-test.scm`
- `set<?` -> `set-lt-p-test.scm`
- `set>=?` -> `set-ge-p-test.scm`
- `set>?` -> `set-gt-p-test.scm`
- `list->set` -> `list-to-set-test.scm`
- `list->set!` -> `list-to-set-bang-test.scm`
- `set->list` -> `set-to-list-test.scm`

### Why
1. `tests/goldfish/liii/set-test.scm` 聚合了大量独立的 set API 测试，不利于维护和定位问题。
2. 拆分为单函数测试文件后，便于单独执行、回归和审查。
3. 与 `string` / `vector` / `base` / `ascii` / `hash-table` / `flexvector` / `enum` 的迁移方式保持一致，目录结构更统一。
4. 每个测试文件只断言对应函数的行为，避免不同 API 的断言互相混杂。

### How
1. 从原 `set-test.scm` 中按函数边界提取测试代码。
2. 按"一个函数一个文件"的方式迁移到 `tests/liii/set/`。
3. 对缺少说明的函数补齐统一格式的文档注释。
4. 保留原有测试覆盖范围，尽量只做结构迁移，不随意改测试语义。
5. 新增 `tests/liii/set-test.scm` 作为迁移后的入口说明文件。
6. 使用项目真实测试入口 `bin/goldfish -m r7rs` 逐个执行 `tests/liii/set/` 下的所有测试文件，确认迁移后的测试可以独立通过。
