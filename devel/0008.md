## 2026/05/03 对比 Chez Scheme 与 Goldfish 的 char-upper-case? 行为差异

### What
使用 Chez Scheme 10.3 遍历全部 Unicode 码位，收集 `char-upper-case?` 返回 `#t` 的字符集合，与 Goldfish 当前实现对比，确认差异范围。所有 Chez Scheme 判定为大写的字符，都需要在测试用例中显式添加。

1. 创建 `compat/char-upper-case-p.scm`，遍历全部 Unicode 码位（0x0000–0xD7FF、0xE000–0x10FFFF），统计 `char-upper-case?` 返回 `#t` 的字符数量及明细。
2. 在 Chez Scheme 10.3（Windows）上执行对比脚本，收集所有满足 `char-upper-case?` 的字符。
3. 初步确认 Goldfish 当前实现的判断范围与 Chez Scheme 存在明显差异，需要调整以匹配 Chez Scheme 的行为。
4. 所有 Chez Scheme 判定为大写的字符，都需要在测试用例中显式添加。
5. `char-upper-case?` 的实现不能依赖 `compat/char-upper-case-p.csv` 数据文件，该文件仅用于对比验证。

### Why

Goldfish Scheme 致力于与主流 Scheme 实现保持行为一致。`char-upper-case?` 是 R7RS 核心过程，其语义直接影响 `(scheme char)` 库以及依赖它的字符串处理功能。当前 C 层实现依赖平台 locale 的 `iswupper`，导致：
- 不同操作系统（Windows/macOS/Linux）下行为不一致；
- 大量 Unicode 大写字符未被正确识别。

### How

1. **先实现完整测试用例**：在 `tests/scheme/char/char-upper-case-p-test.scm` 中写入全部 Chez Scheme 判定为大写的字符的 `check` 断言。此时 C 层实现尚未移除，测试可以验证当前行为与 Chez 的差异。
2. **再实现 Scheme 层 `char-upper-case?`**：在 `goldfish/scheme/char.scm` 中基于 Unicode 码位范围重写 `char-upper-case?`，用 `cond` + `char->integer` 覆盖 C 层的同名绑定。运行测试验证全部通过。
3. **最后移除 C 实现**：清理 `src/s7_scheme_char.c`、`src/s7_scheme_char.h`、`src/s7.c` 中所有与 `char-upper-case?` 相关的 C 代码、静态表和注册逻辑。再次运行测试确认无回归。
