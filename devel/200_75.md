## 2026/04/23 让 char? 支持中文

### What
让 `char?` 谓词能够正确识别中文字符，为后续完整的中文 char 支持奠定基础。

1. 修改字符字面量解析器，使 `#\中` 等 Unicode 字符能被正确解析为字符类型
2. 修改 `char?` 实现，使其对 Unicode 字符返回 `#t`
3. 在 `tests/scheme/base/char-p-test.scm` 中添加中文测试用例

### Why
- 当前 `char?` 仅支持 ASCII 字符（0-255）
- `(char? #\中)` 返回 `#f`，中文字符字面量被解析为 `undefined`
- `(integer->char 20013)`（中字的 Unicode 码点）报 out-of-range 错误

### How

#### 当前 `#\中` 和 `char?` 的代码路径

**`#\中` 解析路径（解析失败 → undefined）：**

1. **Reader 入口**：S7 reader 遇到 `#`，调用 `read_sharp()`（s7.c 第 85302 行）
2. **Token 识别**：`read_sharp()` 读取下一个字符是 `\`，返回 `token_sharp_const`
3. **常量解析入口**：`read_sharp_const()`（第 85694 行）调用 `port_read_sharp()`，即 `file_read_sharp()`（第 22904 行）
4. **名称读取**：`file_read_sharp()` 调用 `file_read_name_or_sharp(sc, port, false)`（第 22871 行）
   - `sc->strbuf[0]` 已经是 `\`（第一个字符在进入此函数前已放入 `strbuf[0]`）
   - 循环读取后续字节：`c = fgetc()`，将字节存入 `sc->strbuf[i++]`
   - 循环条件：`c != EOF && char_ok_in_a_name[c]`
   - `char_ok_in_a_name` 对 128-255 的所有字节默认为 `true`
   - 中文字符 "中" 的 UTF-8 编码为 `E4 B8 AD`，三个字节都 >= 128，因此都被允许读取
   - 循环读取完整 `\中`（即 `\xE4\xB8\xAD`）到 `sc->strbuf`，直到遇到空白或分隔符停止
5. **Sharp 常量创建**：`make_sharp_constant(sc, sc->strbuf, ...)`（第 14877 行）
   - `name` 是 `sc->strbuf` 中的字符串 `\中`（即 `\xE4\xB8\xAD`，长度为 4 字节）
   - 进入 `case '\\':`（第 14946 行）
   - `name[2] == 0` 为 false（因为 `name` 长度为 4）
   - `switch(name[1])`：`name[1]` 是 `0xE4`，不匹配任何已知 case（'n', 'a', 'b', 'd', 'e', 'l', 'r', 's', 't', 'x' 等）
   - 跳出 switch，到达 `return(unknown_sharp_constant(sc, name, NULL));`（第 14996 行）
6. **未知常量处理**：`unknown_sharp_constant()`（第 14821 行）
   - `name` 不以 `<` 开头/结尾，不是 `#<...>` 格式
   - 返回 `make_undefined(sc, name)`（第 14870 行）
7. **结果**：`#\中` 被解析为 `undefined?` 对象

**`char?` 求值路径（返回 #f）：**

1. 用户调用 `(char? #\中)`，参数是 `undefined?` 对象
2. 进入 `g_is_char()`（s7_scheme_char.c 第 295 行）
3. `s7_is_character(obj)` → `is_character(obj)` → `type(obj) == T_CHARACTER`
4. `undefined?` 对象的类型是 `T_UNDEFINED`，不是 `T_CHARACTER`
5. 返回 `s7_f(sc)`，即 `#f`

#### 当前限制
- 字符结构体（`s7.c` 第 845-850 行）中 `c` 字段为 `uint8_t`，只能存储 0-255
- 256 个字符对象预分配在 `chars` 数组中（`s7.c` 第 20284-20316 行）
- `s7_make_character(sc, uint8_t c)` 直接返回 `chars[c]`
- `#\` 字面量解析器（`make_sharp_constant`，第 14946-14995 行）只支持单字节字符或 `\x<hex>`（hex < 256）

#### 实现计划（分步，每步可验证）

**第 1 步：扩展字符结构体和 API（验证：编译通过 + 现有测试不受影响）**
1. 修改字符结构体 `chr`（s7.c 第 845-850 行）：
   - `c` 和 `up_c` 从 `uint8_t` 扩展为 `uint32_t`
   - 其他字段不变
2. 修改 `s7_make_character()`（s7.c 第 20320 行）：
   - 参数从 `uint8_t c` 改为 `uint32_t c`
   - `c < 256` 时：继续返回 `chars[c]`（预分配对象，向后兼容）
   - `c >= 256` 时：使用 `new_cell()` 动态分配字符对象，设置 `T_CHARACTER | T_IMMUTABLE`，填充 `c` 字段
   - **注意**：此步骤暂不实现缓存，同一 codepoint 可能产生多个不同对象（后续第 3 步解决）
3. 修改 `s7_character()`（s7.c 第 20324 行）：返回类型从 `uint8_t` 改为 `uint32_t`
4. 同步修改 `src/s7.h`（第 272-274 行）中的函数签名声明
5. **验证**：`xmake b goldfish` 编译通过，`bin/gf test` 全量测试通过

**第 2 步：修改 `#\` 解析器支持 Unicode + 让 `char?` 识别 Unicode（验证：测试通过）**
1. 修改 `make_sharp_constant()`（s7.c 第 14877 行）中的 `case '\\':` 分支：
   - 在现有的 `name[2] == 0`（单字节字符）和特殊名称（如 `newline`、`space`）处理之后、默认失败之前：
   - 检查 `name` 是否为一个有效的多字节 UTF-8 序列（长度 2-4 字节）
   - 如果是有效的 UTF-8，将其解码为 Unicode codepoint，调用 `s7_make_character(sc, codepoint)` 返回
2. 同步扩展 `#\x<hex>` 分支（第 14971 行）：
   - `new_int` 的上限从 256 提升到 0x10FFFF
   - 调用 `s7_make_character(sc, new_int)` 返回
3. `s7_is_character()`（s7.c 第 20322 行）通过检查 `type(p) == T_CHARACTER` 工作，第 1 步中 `new_cell()` 分配的 Unicode 字符对象已正确设置 `T_CHARACTER` 标记，因此 `char?` 无需额外修改即可识别 Unicode 字符
4. 在 `tests/scheme/base/char-p-test.scm` 末尾追加 Unicode 测试：
   ```scheme
   ;; Unicode 字符测试
   (check (char? #\中) => #t)
   (check (char? #\x4E2D) => #t)
   (check (char? #\🐟) => #t)
   (check (char? 123) => #f)
   (check (char? "中") => #f)
   ```
5. **验证**：`bin/gf test` 全量测试通过
   - 注意：此时 `(eq? #\中 #\x4E2D)` 可能为 `#f`，因为每个字面量会创建独立对象

**第 3 步：添加 Unicode 字符缓存（验证：eq? 测试通过）**
- 问题：第 2 步每次解析 `#\中` 都会调用 `new_cell()` 创建新对象，导致 `(eq? #\中 #\x4E2D)` => `#f`
- 方案：为 Unicode 字符（codepoint >= 256）添加哈希表缓存，确保同一 codepoint 只创建一个对象
- 在 `tests/scheme/base/char-p-test.scm` 中追加缓存测试：
  ```scheme
  ;; Unicode 字符缓存测试（同一 codepoint 应为同一对象）
  (check (eq? #\中 #\x4E2D) => #t)
  ```
- **验证**：`bin/gf test` 全量测试通过

#### 约束
- **不修改 `string-ref`**：`(string-ref "中" 0)` 仍返回 UTF-8 首字节（228），保持字节级语义
- 后续会单独在 `(liii unicode)` 库中提供 `utf8-string-ref`，用于按 Unicode 字符位置访问字符串
- 第一步不涉及 `char->integer` 和 `integer->char` 的改造，仅让 `char?` 能识别中文
