## 2026/05/05 定位并优化 gf fmt 性能瓶颈

### What
通过添加 `(liii logging)` 日志，定位并修复 `gf fmt` 在格式化较大文件时的严重性能问题。

1. 在 `goldfmt-scan.scm`、`goldfmt-format.scm`、`goldfmt.scm` 中添加日志和计数器，分阶段计时
2. 发现 `gf fmt goldfish/scheme/char.scm` 耗时 **约 129 秒**，其中 `rewrite-reader-literals` 的字符串拼接是主要瓶颈
3. 定位到 `rewrite-reader-literals` 使用 `(string-join (reverse result) "")` 进行拼接，而 SRFI-13 的 `string-join` 基于递归 `string-append`，对大列表是 **O(n²)** 复杂度
4. 引入 `fast-string-join` 函数，使用 `open-output-string` + `display` 实现 **O(n)** 拼接
5. 将 `(string-join (reverse result) "")` 替换为 `(fast-string-join (reverse result))`
6. 移除所有调试日志和计数器代码
7. 用 `bin/gf fmt` 格式化修改后的文件

### Why
`goldfish/scheme/char.scm` 包含大量 Unicode 字符定义和中文注释，`rewrite-reader-literals` 需要逐字节扫描并收集成千上万个单字符字符串片段。`string-join` 的 O(n²) 特性导致在处理这类文件时出现灾难性性能下降。

### How
`fast-string-join` 利用 Scheme 的 output string port：

```scheme
(define (fast-string-join strings)
  (let ((out (open-output-string)))
    (let loop ((rest strings))
      (if (null? rest)
        (get-output-string out)
        (begin
          (display (car rest) out)
          (loop (cdr rest)))))))
```

`display` 到 output port 是 O(1) 追加（底层动态扩容），整体遍历一次列表即可完成拼接，复杂度为 O(n)。

**注意**：在优化过程中曾尝试用 `write-char` 逐字符写入 port，但 S7 的 `string-ref` 对 UTF-8 字符串按字节索引返回字节值，`write-char` 会将其解释为 Latin-1 字符并重新编码为 UTF-8，导致中文注释乱码。最终方案保持原有的字符串列表收集方式，仅替换最后的拼接函数。

**性能对比**：
- 优化前：`real 2m9.923s`
- 优化后：`real 约 1.3s`
- 提升幅度：**约 100 倍**

**验证**：
- `bin/gf fmt goldfish/scheme/char.scm` 幂等性通过（第二次运行 "Files updated: 0"）
- `bin/gf tests/.../scan-test.scm`、`scan-string-test.scm`、`scan-file-test.scm` 通过

## 2026/05/05 定位并修复字符字面量扫描问题
### What
修复 `gf fmt` 格式化 `#\六` 和 `#\八` 等 UTF-8 中文字符字面量时产生乱码的问题。

1. 在 `goldfmt-scan.scm` 中添加 `(liii logging)` 日志，定位 `rewrite-reader-literals` 处理字符字面量的过程
2. 发现 `find-char-literal-end` 使用 `reader-delimiter?` 判断字符边界时，由于 `char-whitespace?` 对 UTF-8 多字节字符的中间字节 0x85 返回 `#t`，导致字符字面量被提前截断
3. 将 `reader-delimiter?` 中的 `(char-whitespace? c)` 替换为显式的 ASCII 空白字符检查，避免依赖 C 库 `isspace` 的 locale 相关行为
4. 移除调试日志代码

### Why
`六` 的 UTF-8 编码为 `E5 85 AD`，`八` 的 UTF-8 编码为 `E5 85 AB`。它们的第二个字节都是 0x85（十进制 133）。在 Windows/MSVC 环境下，`char-whitespace?` 底层调用 C 库的 `isspace`，而默认 locale 将 0x85（NEL，Next Line）识别为空白字符。这导致 `find-char-literal-end` 在扫描到该字节时误认为遇到了 delimiter，从而将 `#\六` 截断为 `#\` + 单字节乱码。

### How
`reader-delimiter?` 原本使用 `(char-whitespace? c)` 来识别空白 delimiter，但 Scheme reader 实际使用的空白 delimiter 仅限于 ASCII 范围内的空格、制表符、换行符和回车符。修复方案是将 `char-whitespace?` 替换为显式检查这四个 ASCII 空白字符：

```scheme
(or (char=? c #\space)
    (char=? c #\tab)
    (char=? c #\newline)
    (char=? c #\return)
    ...)
```

这样就不会误将 UTF-8 多字节字符的中间字节（如 0x85）识别为 delimiter。
