## 2026-04-04 修复 tests/scheme/base/ 目录下 import 问题

### What
批量修复 `tests/scheme/base/` 目录下 122 个测试文件的 import 问题：

1. **改为每行一个 import 格式**
   - 原格式：`(import (liii check) (liii base) (liii list) ...)`
   - 新格式：每行一个 import，便于 AI 和人类修改
   ```scheme
   (import (liii check))
   (import (scheme base))
   ```

2. **移除不必要的依赖**
   - `(liii base)` - 不需要，使用 `(scheme base)` 即可
   - `(liii list)` - 不需要
   - `(liii case)` - 不需要
   - `(liii error)` - 不需要
   - `(liii os)` - 不需要

3. **保留必要的依赖**
   - `(liii check)` - 测试框架（必需）
   - `(scheme base)` - 被测试的函数来源（必需）
   - `(scheme file)` - read-line 测试需要（已保留）
   - `(liii list)` - 以下测试需要：append-test, lambda-test, length-test, map-test, memv-test, reverse-test, set-cdr-bang-test（使用 filter, last-pair 等函数）
   - `(liii base)` - 以下测试需要：and-test, boolean-eq-p-test, floor-quotient-test, truncate-slash-test（使用 and-let*, receive 等函数）
   - `(liii error)` - exact-integer-sqrt-test 需要（使用 type-error）

### Why
1. 每行一个 import 格式更简洁，便于后续批量修改和维护
2. 移除明显不必要的依赖（如大量文件多余的 liii base, liii list 等）
3. 保留实际需要的依赖，确保测试能正确运行

### How
使用 Python 脚本批量处理：
1. 遍历所有 `tests/scheme/base/*.scm` 文件
2. 提取 import 块中的模块
3. 过滤掉不必要的 liii 模块
4. 将每个模块转换为单独的一行 import
5. 将缺失 `(scheme base)` 的文件补全

### 文件命名规则
本次迁移沿用统一的特殊符号转写规则：

1. `?` -> `-p`
   - `eqv?` -> `eqv-p-test.scm`
   - `string?` -> `string-p-test.scm`

2. `!` -> `-bang`
   - `string-set!` -> `string-set-bang-test.scm`
   - `bytevector-u8-set!` -> `bytevector-u8-set-bang-test.scm`

3. `->` -> `-to-`
   - `number->string` -> `number-to-string-test.scm`
   - `string->symbol` -> `string-to-symbol-test.scm`

4. `=` -> `-eq`
   - `boolean=?` -> `boolean-eq-p-test.scm`

5. `*` -> `-star`
   - `*` -> `star-test.scm`

6. `/` -> `-slash-`
   - `truncate/` -> `truncate-slash-test.scm`
   - `/` -> `slash-test.scm`

7. 关系运算符中的 `<` / `>` / `<=` / `>=` 按语义写法转写
   - `<` -> `lt`
   - `>` -> `gt`
   - `<=` -> `le`
   - `>=` -> `ge`

8. 单独的算术运算符名称采用语义化命名
   - `+` -> `plus-test.scm`
   - `-` -> `minus-test.scm`
   - `*` -> `star-test.scm`
   - `/` -> `slash-test.scm`

### Why
1. `tests/goldfish/liii/base-test.scm` 文件体量过大，维护和定位失败用例都比较困难。
2. 将测试按函数拆分后，可以更方便地单独执行、回归与审查。
3. 与最近的 `string` / `vector` / `ascii` 迁移方式保持一致，目录结构更统一。
4. 为每个测试文件补齐统一格式的文档注释，便于后续维护和继续拆分。

### How
1. 盘点 `base-test.scm` 中的文档块与对应测试区域，识别可直接“一函数一文件”迁移的部分。
2. 将带文档块的函数测试批量迁移到 `tests/liii/base/`。
3. 对 `equal?` 后面那段没有独立文档块的语法/高阶函数测试，额外拆成 `lambda`、`map`、`if`、`cond`、`case` 等文件。
4. 为每个新文件补齐统一的文档块注释，并沿用统一的文件命名规则。
5. 删除旧的聚合测试文件 `tests/goldfish/liii/base-test.scm`。
6. 新增 `tests/liii/base-test.scm`，仅用于说明测试已迁移到目录中。
7. 使用项目真实测试入口 `bin/goldfish -m r7rs` 逐个执行 `tests/liii/base/` 下的所有测试文件，确认迁移后全部通过。
