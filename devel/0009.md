## 2026/05/07 规划修复方案

### What

1. `gf fix` 在处理 `#(` 向量开括号时，仅将 `(` 的缩进作为匹配基准，导致 `)` 即使对齐到 `#` 的位置仍被判定为缩进不匹配。
2. 当 `)` 后面带有 `;#` 注释（用于标记 vector 结束）时，`gf fix` 删除原 `)` 并在 `last-code-end-offset` 处插入新 `)`，导致 `;#` 注释被孤立在单独一行，语义丢失。
3. 具体受影响文件包括 `tests/liii/json/json-to-string-test.scm` 等使用 `#(` 和 `) ;#` 风格的测试文件。

### Why

`#(` 是 Scheme 向量字面量的标准语法。在 Goldfish 代码库中，开发者常用 `) ;#` 注释来明确标记向量的结束边界，以提高深层嵌套数据的可读性。`gf fix` 作为括号修正工具，必须识别并尊重这种编码约定，避免"修正"破坏代码结构和注释语义。

### How

修改 `tools/fix/liii/goldfix-repair.scm`：

1. 修改 `make-open-frame-from-token`，增加 `prev-token` 参数。
2. 在 `process-line!` 和 `process-token!` 中传递前一个 token。
3. 当遇到 `open-paren` 且前一个 token 是紧邻的 `#` 时，将 `open-frame` 的 `column` 设为 `#` 的 column，而非 `(` 的 column。
4. 这样，`#(` 对应的闭括号 `)` 可以合法地对齐到 `#` 所在的列，不会被误判为 `indent-mismatch-close`。
5. 同时，由于 `)` 不再被删除/移动，`;#` 注释自然保持原有的 `) ;#` 形式。

核心改动示意：

```scheme
(define (make-open-frame-from-token token prev-token)
  (let ((is-vector-prefix
         (and prev-token
              (eq? (fix-token-type prev-token) 'other)
              (string=? (fix-token-text prev-token) "#")
              (= (fix-token-end prev-token) (fix-token-offset token)))))
    (make-open-frame :offset
      (if is-vector-prefix
        (fix-token-offset prev-token)
        (fix-token-offset token))
      :line (fix-token-line token)
      :column (if is-vector-prefix
                (fix-token-column prev-token)
                (fix-token-column token))
      :tag-name ""
    ) ;make-open-frame
  ) ;let
) ;define
```

在 `process-line!` 中：

```scheme
(let loop
  ((tokens (fix-line-tokens line)) (prev-token #f))
  (if (not (null? tokens))
    (begin
      (process-token! (car tokens) line prev-token)
      (loop (cdr tokens) (car tokens))
    ) ;begin
  ) ;if
) ;let
```

并相应修改 `process-token!` 的签名以接收 `prev-token`。
