# [200_63] 命令行工具注册系统规范

## 背景

目前 `gf` 的子命令 `doc`、`source` 和 `help` 是在 C++ 代码中硬编码的。为了支持通过 Scheme 实现更多的子命令，需要设计一个工具注册系统。

本规范首先实现 `help` 子命令，用于显示所有已注册工具的帮助信息。

## 设计目标

1. 通过配置文件动态注册工具
2. 统一工具加载逻辑
3. 支持当前仓库覆盖内置工具
4. 工具路径只在执行时动态设置，不影响当前仓库开发

## 目录结构规范

```
gfproject.json              # 项目配置文件（包含 tools 定义）
tools/
└── help/                   # help 子命令
    └── liii/
        └── goldhelp.scm
```

### 目录命名规则

- 工具根目录名：与注册表中的子命令名一致（如 `doc`, `source`）
- 组织目录：与 `organization` 字段一致（如 `liii`）
- Scheme 文件命名：`{organization}{module}*.scm`（如 `golddoc.scm`, `golddoc-cli.scm`）

### 工具搜索路径（按优先级）

1. **当前仓库工具目录**: `<cwd>/tools/<command>/<organization>/`
2. **内置工具目录**: `<gf_lib>/tools/<command>/<organization>/`

## gfproject.json 规范

### 文件位置

- 内置配置：`<gf_lib>/gfproject.json`（定义 gf 内置的工具）
- 当前仓库配置：`<cwd>/gfproject.json`（可选，用于扩展或覆盖内置工具）

当前仓库的 `gfproject.json` 可以定义额外的工具或覆盖内置工具的行为。

### 格式定义

```json
{
  "tools": {
    "<command>": {
      "organization": "<org-name>",
      "module": "<module-name>",
      "description": {
        "<lang>": "<description>"
      }
    }
  }
}
```

**说明**：`organization` 和 `module` 是可选字段。如果两者都缺失，表示该工具只提供了描述信息（用于 help 显示），没有 Scheme 实现。如果两者都存在，表示这是一个完整的工具实现。

### 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `tools` | object | 所有注册的工具，key 为子命令名 |
| `organization` | string | （可选）组织命名空间，对应目录结构中的中间层。与 `module` 同时存在时表示有 Scheme 实现 |
| `module` | string | （可选）模块名，对应 `(liii golddoc)` 中的 `golddoc` 部分。与 `organization` 同时存在时表示有 Scheme 实现 |
| `description` | object | 工具的多语言描述，key 为语言代码（`en_US`, `zh_CN`），value 为对应语言的描述。当前默认显示 `en_US` |

### 示例

只有描述信息（无 Scheme 实现）：

```json
{
  "tools": {
    "doc": {
      "description": {
        "en_US": "Browse Goldfish Scheme library documentation",
        "zh_CN": "浏览 Goldfish Scheme 库文档"
      }
    }
  }
}
```

完整工具定义（有 Scheme 实现）：

```json
{
  "tools": {
    "help": {
      "organization": "liii",
      "module": "goldhelp",
      "description": {
        "en_US": "Display help information",
        "zh_CN": "显示帮助信息"
      }
    },
    "source": {
      "organization": "liii",
      "module": "goldsource",
      "description": {
        "en_US": "View source code of library functions",
        "zh_CN": "查看库函数源代码"
      }
    }
  }
}
```

## 模块规范

### 模块名格式

模块完整名称为 `(organization module)`，如 `(liii golddoc)`。

### 入口函数

每个工具模块必须导出一个 `main` 函数，作为子命令的入口点。

```scheme
(define-library (liii golddoc)
  (import ...)
  (export main)
  (begin
    (define (main)
      ;; 工具逻辑
      )))
```

入口函数无需显式返回退出码，正常运行时默认返回 0。

## 关于 fix 命令

`fix` 命令保持现有实现不变，不使用本规范定义的工具注册系统。

## 2026/04/12 修复 xmake.lua 打包配置

### What
修复 `xmake.lua` 中的 `add_installfiles` 配置，解决 v17.11.47 打包漏文件问题：

1. 添加 `gfproject.json` 到安装文件列表
2. 修复 tools 目录路径（`goldtest`→`test`, `golddoc`→`doc`）
3. 添加缺失的 `tools/help/` 和 `tools/version/` 目录
4. 使用 `(tools/**)` 通配符打包整个 tools 目录，避免以后遗漏新文件

### Why
v17.11.47 打包后的版本所有子命令都无法使用，因为：
- `gfproject.json` 未被包含在打包文件中
- tools 目录结构变更后（`goldtest`→`test` 等），xmake.lua 未及时更新
- 新添加的 `help` 和 `version` 工具未添加到打包配置

### How
简化配置，使用 `(tools/**)` 通配符一次性打包整个 tools 目录，同时保留 `gfproject.json` 的单独配置。

## 2026/04/12 修复工具搜索路径

### What
修复 `find_tool_root_by_command` 函数，添加对当前目录下 `tools/<command>` 的支持。

1. 优先搜索 `<cwd>/tools/<command>`
2. 其次搜索 `<gf_lib>/tools/<command>`
3. 最后搜索 `<gf_lib>/../tools/<command>`

### Why
之前的代码虽然文档中说明了支持当前仓库工具目录，但实际 `find_tool_root_by_command` 函数并未实现此功能，导致用户无法在当前项目目录下创建自定义工具。

### How
修改 `find_tool_root_by_command` 函数，首先获取当前工作目录，将其与 `tools/<command>` 拼接后加入搜索路径候选列表的最前面。
