## 2026/06/24 厘清 fix 职责、移除隐式 fmt 与 exclude

### What
1. `fix-file` / `fix-directory` 不再先 `os-call` 一次 `gf fmt`，fix 只做括号修正
2. 移除 `gf fix` 的 `--exclude` 参数，以及 `file-excluded?`、`normalize-sep`、`parse-excludes`、`exclude-entry->path`、`parse-exclude-json`、`read-project-excludes` 全部相关代码；fix 不再读取 `gfexclude.json`
3. import 去掉仅 exclude 用到的 `(liii json)`
4. `--help` 移除 `--exclude` 说明，新增「fix 依赖规整缩进推断括号结构，建议先 `gf fmt`」提示
5. `repair-source` 增加缩进不可信守卫：当输入括号已平衡（`parentheses-balanced?` 为 `#t`）但 `repair-parentheses` 仍产生了 edit 时，判定缩进不可信，**原样返回**并打印「请先运行 gf fmt」提示，不再静默破坏代码；并 export `repair-source`
6. 新增 `goldfix-test.scm` 覆盖守卫：平衡+缩进乱原样返回、平衡+缩进正常不变、缺括号/多括号正常修复
7. fix 输出区分两种「未改动」：本就平衡 → `Already balanced:`；守卫拦截 → 警告 + `Skipped (run gf fmt first):`；缺括号 → `Updated:`。`fix-file-core` 返回值由 `#t/#f/'cached` 改为符号 `'updated/'balanced/'skipped/'cached`，靠 `repair-source*` 的第二返回值 `skipped?` 区分

### Why
之前 `gf fix` 在修正括号前会隐式先跑一遍 `gf fmt`，职责分割不清晰。调查发现 `repair-parentheses` 是**缩进驱动**的：`close-by-indent-boundary!` / `confirm-pending-closes!` / `indent-mismatch-close` 都靠 token 的列位置（缩进）推断括号结构——其注释本身就写明「根据可信缩进修正缺失或多余的右括号」。这意味着 fix 必须有规整缩进作为前提。

实测：括号本已平衡、但缩进错乱的代码
```
(define (f x)
(let ((y 2))
(+ x y)))
```
会被 fix 按缩进误判成三个各自平衡的表达式
```
(define (f x))
(let ((y 2)))
(+ x y)
```
因此 fix 对 fmt 是**强依赖**。仅去掉 fmt 调用并不可行——那样 fix 会变本加厉地破坏代码。最终方案分两步：
- **职责单一**：fix 不再隐式调 fmt，`--help` 提示用户先手动 `gf fmt`；并移除 fix 用不到的 `--exclude` / `gfexclude.json`
- **安全守卫**：即使有用户忘了先 fmt，fix 也要能「拒绝破坏」而不是静默改坏。最可靠的不可信信号是「括号已平衡却要改动」——平衡的代码任何按缩进的调整都只会破坏结构。`repair-source` 检测到此情形时原样返回并提示，缺括号/多括号（真正需要修的）不受影响。

### How
- **去 fmt、去 exclude**：fix 主流程（`fix-file` / `fix-file-list` / `fix-directory` / `fix-changed-since`）逐层去掉 `os-call fmt` 与 `excludes` 形参、`file-excluded?` 过滤分支；`main` 的 cond 分支同步收窄；删除孤立的 exclude 辅助函数群（`normalize-sep` / `parse-excludes` / `parse-exclude-json` / `read-project-excludes` 等）及随之失效的 `(liii json)` import
- **安全守卫**：守卫放在 `repair-source*`（CLI 入口包装层）而非底层 `repair-parentheses` / `fix-string`。判定条件为 `(and (parentheses-balanced? source) (not (null? edits)))`——括号已平衡却产生了 edit，即缩进不可信，此时原样返回 `source` 并打印「请先运行 gf fmt」。缺括号/多括号（真正需要修的）`parentheses-balanced?` 为 `#f`，不进守卫，正常修复。`repair-source*` 返回 `(values repaired skipped?)`，`skipped?` 把"是否被守卫拦截"的信号穿透给上层
- **区分输出**：`fix-file-core` 返回值由布尔改为符号 `'cached`/`'updated`/`'balanced`/`'skipped`（用 `skipped?` 区分后两种"未改动"）。`fix-file-list` / `fix-directory` / `main` 的 cond 据此精确匹配符号打印：`Updated:` / `Already balanced:` / `Skipped (run gf fmt first):`。目录模式对 `balanced` 静默（避免刷屏），单文件模式全部打印
- **测试落点**：守卫与结果区分测试放在针对 `(liii goldfix)` 主模块的新文件 `goldfix-test.scm`，覆盖 `repair-source`（4 种输入）与 `repair-source*`（3 种 `skipped?` 信号）；为此 export `repair-source` / `repair-source*`
