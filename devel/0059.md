## 2026-05-28 liii_subprocess 接口设计

### What

设计 Goldfish Scheme 的 subprocess 库，用于替代 shell 脚本。

### Why

Goldfish Scheme 目前缺少一个简洁、统一的 subprocess 调用接口。现有方案要么过于底层（直接暴露 fork/exec），要么模仿 Python subprocess 导致 API 过于复杂。我们希望提供一个**比 Python 更简洁**的库，用三个基础函数覆盖 80% 场景，再通过组合设施替代 shell 脚本中的控制流。

### How

- **define* + keywords** 风格：options 用 `:cwd`、`:env` 等 keyword 参数传递，比 alist 更简洁
- **三层架构**：基础原子层（`run` / `run-string` / `run-values`）+ 基础设施层（`run-and` / `run-pipe` 等组合逻辑）+ 隐藏的系统调用层
- **符号命令 + 注册表**：列表形式的第一个元素可以是符号，通过 `run-set!` / `run-get` / `run-allow!` / `run-ban!` 全局定制命令解析，支持 lambda 注册自定义命令
- **IO 分离设计**：管道用专门的 `run-pipe`，输出重定向用 `:stdout`/`:stderr`，不依赖 shell 解析 `|` 和 `>`

**实现顺序（TDD，每个 Stage 先写测试再写实现）：**

1. **Stage 1 — `run` 基础版本（纯 Scheme）**
   - 先写 `tests/liii/subprocess/run-test.scm`
   - 基于 `os-call` + `chdir` 实现，不写 C++
   - 仅支持 `:cwd`，其他 keyword 留空占位
   - 字符串 command 直接透传，`'("make" "clean")` 列表形式用 `string-join` 拼接

2. **Stage 2 — 符号命令管理（纯 Scheme）**
   - 先写 `tests/liii/subprocess/run-set-bang-test.scm`、`run-get-test.scm`、`run-allow-bang-test.scm`、`run-ban-bang-test.scm`
   - `run-set!` / `run-get`：基于 `hash-table` 实现注册表
   - `run-allow!` / `run-ban!`：基于列表实现白名单/黑名单检查
   - `run` 增加对符号命令的解析分支

3. **Stage 3 — `run-string` / `run-values` + 完整 keyword 支持**
   - 先写各函数的 `-test.scm`
   - 基于 `~/git/tbox` 实现，利用 tbox 的进程和管道封装
   - `run-string` 返回 stdout 字符串，`run-values` 多值返回 stdout、stderr、exit-code
   - 同步支持 `:env`、`:input`、`:timeout`、`:stdout`、`:stderr`、`:stdin`

4. **Stage 4 — 基础设施层**
   - 先写各组合函数的 `-test.scm`
   - `run-and` / `run-or` / `run-sequence`
   - `run-pipe`
   - `run-if` / `run-when`
   - 全部由 `run` / `run-string` / `run-values` 组合而成，纯 Scheme
   - `:timeout` 为整体超时（整个组合序列的总时间限制），可借助 `(liii timeit)` 的计时机制实现
   - 性能测试可使用 `(liii timeit)`

# Spec

## 1. 设计原则

- **函数+参数风格**，不采用面向对象风格
- **比 Python 更简洁**，三个基础函数覆盖 80% 场景
- **上层基础设施**，基于基础函数组合出控制流设施
- **不暴露底层系统调用**（fork/execvp/pipe/wait），由实现层处理

## 2. 基础层

### 2.1 `run` — 仅执行，不捕获输出

```scheme
(run command)
(run command keyword value ...)
```

| 参数 | 类型 | 说明 |
|------|------|------|
| `command` | 字符串或列表 | `"make clean"` 或 `'(make "clean")` |
| `keyword value ...` | keyword 参数（可选） | 见下方通用 options |

**返回值：** 退出码（整数，0 表示成功）

```scheme
(run "make clean")
(run '(go "build") :cwd "./cmd/app")
```

---

### 2.2 `run-string` — 执行并返回 stdout 字符串

```scheme
(run-string command)
(run-string command keyword value ...)
```

**返回值：** `Either` 类型
- 成功时返回 `Right`，内含标准输出字符串（末尾换行保留）
- 失败时返回 `Left`，内含退出码（整数）

```scheme
(run-string '(git "rev-parse" "--short" "HEAD"))
;; => Right "a1b2c3d\n"

(either-right? (run-string "echo hello"))  ;; => #t
(to-right (run-string "echo hello"))       ;; => "hello\n"
(either-left? (run-string "false"))        ;; => #t
(to-left (run-string "false"))             ;; => 1
```

---

### 2.3 `run-values` — 执行并多值返回全部信息

```scheme
(run-values command)
(run-values command keyword value ...)
```

**返回值（values）：**
1. stdout 字符串
2. stderr 字符串
3. 退出码（整数）

```scheme
(let-values (((out err code) (run-values '(go "test"))))
  (if (zero? code)
      (display out)
      (begin
        (display "error: ")
        (display err))))
```

---

### 2.4 符号命令管理

符号命令（见 3.2.3）的解析依赖全局注册表，以下函数用于管理该注册表。

#### 2.4.1 `run-set!` — 注册符号命令路径

```scheme
(run-set! symbol path)
```

为符号 `symbol` 注册解析方式。值可以是字符串、path 对象或 lambda：

- **字符串或 path 对象**：指向外部可执行文件
- **lambda**：注册自定义命令，调用时直接执行该过程

```scheme
;; 注册外部可执行文件
(run-set! 'ls "/bin/ls")
(run-set! 'tsc (path "./node_modules/.bin/tsc"))

;; 注册自定义 lambda
(run-set! 'hello (lambda () (display "hello world\n")))
(run '(hello))   ;; => hello world
```

---

#### 2.4.2 `run-get` — 查询符号命令路径

```scheme
(run-get symbol)
```

返回符号 `symbol` 对应的值（字符串注册值返回时自动转为 path 对象，lambda 原样返回）。未注册且无法通过默认规则解析的符号返回 `#f`。

```scheme
(run-set! 'ls "/bin/ls")
(run-get 'ls)        ;; => <path> 对象
(run-get 'not-exist) ;; => #f
```

---

#### 2.4.3 `run-allow!` — 设置命令白名单

```scheme
(run-allow! symbol)
(run-allow! symbol-list)
```

只允许指定的符号命令执行。参数可以是单个符号或符号列表。白名单模式下，未列入的符号命令执行时报错。

```scheme
(run-allow! '(git ls))
(run '(git "status"))    ;; 正常执行
(run '(rm "-rf" "/"))    ;; => 抛出错误：符号命令 'rm 不在白名单中

;; 取消白名单
(run-allow! '())
```

---

#### 2.4.4 `run-ban!` — 禁止符号命令

```scheme
(run-ban! symbol)
```

显式禁止某个符号命令。禁止后该符号无法解析，执行时报错。

**对字符串命令同样生效**：若字符串命令以被禁止的符号名称开头（后跟空格或结束），也会报 `value-error`。

**默认禁止：** `'rm` 默认已在黑名单中。

```scheme
(run-ban! 'rm)
(run '(rm "-rf" "/"))      ;; => 抛出错误：符号命令 'rm 已被禁止
(run "rm -rf /tmp/test")   ;; => 抛出错误：Command 'rm -rf /tmp/test' has been banned
```

---

#### 2.4.5 `run-unban!` — 取消禁止符号命令

```scheme
(run-unban! symbol)
```

从黑名单中移除指定符号，恢复其可执行权限。

```scheme
(run-ban! 'echo-cmd)
(run '(echo-cmd "hello"))  ;; => 报错
(run-unban! 'echo-cmd)
(run '(echo-cmd "hello"))  ;; => 正常执行
```

## 3. 通用 options

所有三个基础函数共享同一套 keyword 参数：

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `:cwd` | 字符串或 path 对象 | 当前目录 | 子进程工作目录。path 对象由 `(liii path)` 提供，内部自动转字符串。命令中显式含 `cd` 时不可同时设置，否则抛 `value-error` |
| `:env` | alist | 继承当前环境 | **完整覆盖**环境变量。如 `'(("FOO" . "bar") ("PATH" . "/usr/local/bin"))` |
| `:input` | 字符串 | `""` | 写入子进程 stdin 的数据 |
| `:timeout` | 整数（秒） | 无 | 超时后发送 SIGTERM；未退出则再发 SIGKILL。超时后底层返回 exit-code = -1 |
| `:stdout` | 符号、字符串或 path 对象 | 继承 | `'discard` 丢弃；字符串或 path 表示写文件 |
| `:stdout-mode` | 符号 | 覆盖 | 需配合 `:stdout` 使用；`'append` 表示追加写入，否则覆盖 |
| `:stderr` | 符号、字符串或 path 对象 | 继承 | 同上；特殊值 `'stdout` 实现 2>&1；`'discard` 丢弃 |
| `:stderr-mode` | 符号 | 覆盖 | 需配合 `:stderr` 使用；`'append` 表示追加写入，否则覆盖 |
| `:stdin` | 符号、字符串或 path 对象 | 继承 | 字符串或 path 表示从文件读取；`'null` 表示空输入 |

```scheme
;; 输入管道 + 超时
(run-values '(grep "pattern")
            :input "hello foo\nbar baz\n"
            :timeout 5)

;; 输出重定向到文件
(run "ls" :stdout "/tmp/list.txt")

;; stderr 合并到 stdout
(run-string "go build" :stderr 'stdout)

;; 追加模式
(run "cmd" :stdout "log.txt" :stdout-mode 'append)
```

## 3.1 IO 设计

### 3.1.1 默认行为

不传入任何 IO 相关参数时，子进程的 stdin/stdout/stderr 继承自父进程：

```scheme
(run "cat")        ;; stdin/stdout/stderr 与当前终端绑定
(run-string "ls")  ;; stdout 被捕获为字符串，stderr 仍输出到终端
```

### 3.1.2 输入（stdin）

| 方式 | 参数 | 说明 |
|------|------|------|
| 继承 | 不传 `:stdin`、`:input` | 子进程从当前终端读取 |
| 字符串注入 | `:input` | 将字符串直接写入子进程 stdin，结束后关闭管道 |
| 文件输入 | `:stdin` 传字符串或 path | 从指定文件读取内容作为 stdin |
| 空输入 | `:stdin 'null` | 等价于 `< /dev/null` |

```scheme
;; 字符串注入
(run-values '(grep "foo") :input "foo bar\nbaz qux\n")

;; 从文件读取 stdin
(run "wc -l" :stdin "/etc/passwd")
```

**`:input` 与 `:stdin` 互斥。** 同时传入时行为未定义，实现层可报错或以 `:input` 优先。

### 3.1.3 输出（stdout / stderr）

| 值 | 语义 |
|----|------|
| 省略（默认） | 继承父进程对应 fd |
| `'discard` | 丢弃输出（等价 `> /dev/null`） |
| 字符串或 path 对象 | 覆盖写入指定文件（配合 `:stdout-mode` 可追加） |
| `'stdout`（仅 `:stderr`） | stderr 合并到 stdout（等价 `2>&1`） |

```scheme
;; 丢弃 stdout
(run "noisy-cmd" :stdout 'discard)

;; 输出写入文件
(run "ls" :stdout "/tmp/list.txt")

;; 追加写入
(run "cmd" :stdout "log.txt" :stdout-mode 'append)

;; 2>&1
(run "cmd" :stdout "/tmp/out.txt" :stderr 'stdout)
```

### 3.1.4 返回值与 IO 的交互

三个基础函数的返回值受 IO 参数影响：

| 函数 | stdout 去向 | stderr 去向 | 返回值 |
|------|------------|------------|--------|
| `run` | 继承或文件 | 继承或文件 | exit code |
| `run-string` | 默认：捕获为字符串；`:stdout` 指定时：按 `:stdout` 处理，对应位置返回空字符串 | 继承或按 `:stderr` 处理 | `Either`：成功为 `Right` 含 stdout，失败为 `Left` 含 exit-code |
| `run-values` | 默认：捕获为字符串；`:stdout` 指定时：按 `:stdout` 处理 | 默认：捕获为字符串；`:stderr` 指定时：按 `:stderr` 处理 | values: stdout-string stderr-string exit-code |

**规则：** 当 `:stdout` 被显式指定（非默认值）时，`run-string` 和 `run-values` 不再将 stdout 作为返回值的一部分，对应位置返回空字符串。`:stderr` 同理。

```scheme
;; stdout 写入文件，run-string 返回空字符串
(run-string "echo hi" :stdout "/tmp/hi.txt")
;; => Right ""

;; run-values 中 stdout 写入文件，stderr 仍捕获
(let-values (((out err code) (run-values "echo hi" :stdout "/tmp/hi.txt")))
  out) ;; => ""
```

## 3.2 command 设计

`command` 参数支持两种形式：

| 形式 | 示例 | 说明 |
|------|------|------|
| 字符串 | `"make clean"` | 由 `/bin/sh -c` 执行，支持通配符、`$VAR` 变量展开等 shell 特性。变量来源包括系统环境变量和 `:env` 传入的值。不支持管道 `\|`，管道请用 `run-pipe` |
| 列表 | `'(make "clean")` | 直接执行目标程序，**不经过 shell**，参数按列表元素逐字传递。列表第一项必须是**符号** |

### 3.2.1 字符串形式

字符串形式直接交给 `/bin/sh -c` 执行，支持通配符、变量展开等 shell 特性：

```scheme
;; 支持通配符、变量展开
(run "ls *.scm")
(run "echo $HOME")

;; :env 传入的变量也可通过 $VAR 展开
(run "echo $CC" :env '(("CC" . "clang")))
```

**注意：** `run` 的字符串形式**不支持管道 `\|`**。管道请使用专门的 `run-pipe`（见 4.2.1）。

**缺点：**
- 存在命令注入风险，拼接用户输入时需格外小心
- 依赖系统中 `/bin/sh` 的实现，可移植性稍差

### 3.2.2 列表形式（推荐）

列表形式的第一个元素必须是**符号**，后续元素为字符串参数。内部直接定位可执行文件并逐字传参，不经过 shell 解析。

```scheme
;; 正确：符号开头
(run '(grep "foo bar" "*.txt"))
(run '(echo "hello; rm -rf /"))

;; 错误：第一项是字符串，会抛 value-error
;; (run '("grep" "foo" "*.txt"))
;; => Command list must start with a symbol, e.g. '(grep "yyy" "zzz"), got: ...
```

**限制：**
- 不支持通配符、`$VAR` 变量展开等 shell 特性（参数逐字传递）
- 管道使用 `run-pipe`，输出重定向使用 `:stdout`/`:stderr`

### 3.2.3 符号命令

列表形式的第一个元素可以是任意符号，通过全局可定制的命令查找器解析为实际可执行文件：

```scheme
;; 符号命令
(run '(ls "-l" "/tmp"))
(run '(git "status"))
```

符号命令不经过 shell，也不直接按文件名查找，而是通过**命令注册表**解析。注册表允许为每个符号单独指定可执行文件路径：

```scheme
;; 注册符号命令（值可以是字符串、path 对象或 lambda）
(run-set! 'ls "/bin/ls")
(run-set! 'tsc (path "./node_modules/.bin/tsc"))

;; 未注册的符号命令按默认规则（symbol->string，如 PATH 查找）解析
(run '(git "status"))
```

注册表是全局的，作用域覆盖所有后续通过符号发起的 `run` / `run-string` / `run-values` 调用。

通过 `run-get` 可查询符号对应的解析结果（字符串注册值返回 path 对象）：

```scheme
(run-set! 'ls "/bin/ls")
(run-get 'ls)        ;; => <path> 对象
```

`run-ban!` 用于显式禁止某个符号命令，禁止后该符号无法解析，执行时报错。对字符串命令同样按前缀匹配生效：

```scheme
(run-ban! 'rm)
(run '(rm "-rf" "/"))      ;; => 抛出错误：符号命令 'rm 已被禁止
(run "rm -rf /tmp")         ;; => 抛出错误：Command 'rm -rf /tmp' has been banned
```

`run-allow!` 设置白名单，只允许指定的符号命令。参数可以是单个符号或符号列表：

```scheme
;; 单个符号
(run-allow! 'git)

;; 多个符号
(run-allow! '(git ls))

(run '(git "status"))    ;; 正常执行
(run '(rm "-rf" "/"))    ;; => 抛出错误：符号命令 'rm 不在白名单中
```

取消白名单限制：

```scheme
(run-allow! '())         ;; 空列表表示取消白名单
```

**使用场景：**
- 需要跨平台或跨环境定制命令路径时
- 项目内部有本地工具链（如 `node_modules/.bin`）时
- 需要限制某些危险命令的执行权限时

### 3.2.4 选择建议

- **默认使用列表形式（符号 + 字符串参数）** — 安全、明确、无注入风险
- **字符串形式用于快速脚本** — 需要 shell 语法（通配符、变量展开）的 adhoc 场景
- **符号形式用于可定制查找** — 需要自定义命令解析规则时使用
- **不要用字符串拼接用户输入** — 需要动态参数时，用列表 + `(cons cmd args)` 组装

```scheme
;; 安全：列表形式 + Scheme 拼接
(let ((pattern (read-line)))
  (run (list 'grep pattern "/etc/passwd")))

;; 危险：字符串拼接用户输入
(let ((pattern (read-line)))
  (run (string-append "grep " pattern " /etc/passwd")))
```

## 4. 基础设施层

基于三个基础函数组合出控制流设施，用于替代 shell 脚本。
所有基础设施函数的返回值均为 `Either` 类型，便于统一错误处理。

### 4.1 顺序控制

#### 4.1.1 `run-and` — 失败即停（替代 `&&`）

```scheme
(run-and command ...)
(run-and command ... keyword value ...)
```

依次执行命令，任一命令 exit code ≠ 0，立即停止并返回该错误信息。全部成功返回最后一个命令的 exit code。

**返回值：** `Either`
- 全部成功：`Right`（内含最后一个命令的 exit code）
- 任一失败：`Left`（内含 `(list code command)`）

末尾的 keyword 参数分两类：

**共享选项**（作用到每个命令）：
- `:cwd`、`:env` — 每个命令都在该工作目录/环境下执行
- `:timeout` — **整体超时**，整个 `run-and` 序列的总时间限制，超时后终止当前命令，后续命令不再执行

**IO 选项**（只对特定命令生效）：
- `:stdout`、`:stderr` — 只对**最后一个命令**生效
- `:input`、`:stdin` — 只对**第一个命令**生效

这样设计更接近 shell `&&` 的直觉：

```scheme
;; 只有 make test 的 stdout 被重定向到文件
(run-and "make clean"
         "make build"
         "make test"
         :stdout "test.log")
```

**注意：** `run-and` 中每个命令都是独立子进程，shell 内建命令（如 `cd`、`export`）的副作用不会影响后续命令。要改变工作目录请用 `:cwd`。

命令中显式包含 `cd` 与 `:cwd` 语义矛盾，**不允许同时使用**，否则抛出 `value-error`：

```scheme
;; 错误：cd 只在独立子进程中生效，ls 仍在原目录
(run-and "cd /tmp" "ls")

;; 正确：用共享选项
(run-and "ls" "pwd" :cwd "/tmp")

;; 报错：命令含 cd 同时又传 :cwd
(run "cd /tmp" :cwd "/home")
;; => 抛出 value-error

;; 正常用法
(run-and "make clean"
         "make build"
         "make test")

(run-and "make clean"
         "make build"
         :cwd "./project" :timeout 300)
```

---

#### 4.1.2 `run-or` — 失败才继续（替代 `||`）

```scheme
(run-or command ...)
(run-or command ... keyword value ...)
```

依次执行命令，任一命令 exit code = 0 立即停止并返回成功。全部失败返回最后一个命令的错误信息。

**返回值：** `Either`
- 任一成功：`Right 0`
- 全部失败：`Left`（内含 `(list code command)`）

```scheme
(run-or '(which "python3")
        '(which "python")
        "echo 'no python found'")
```

---

#### 4.1.3 `run-sequence` — 不管成败都跑完（替代 `;`）

```scheme
(run-sequence command ...)
(run-sequence command ... keyword value ...)
```

依次执行所有命令，**不因为失败而中断**。返回最后一条命令的 exit code。

**返回值：** `Either`
- 最后一条成功：`Right`（内含最后一条命令的 exit code）
- 最后一条失败：`Left`（内含最后一条命令的 exit code）

```scheme
(run-sequence "echo 'start'"
             '(false)          ;; 失败但不阻断
             "echo 'end'")
```

---

### 4.2 管道（替代 `|`）

#### 4.2.1 `run-pipe`

```scheme
(run-pipe command ...)
(run-pipe command ... keyword value ...)
```

将多个命令串联为管道，前一个命令的 stdout 作为后一个命令的 stdin。

**返回值：** `Either`
- 全部成功：`Right`（内含管道最后一级的 stdout 字符串）
- 任一失败：`Left`（内含 `(list code command)`）

```scheme
(run-pipe "cat /etc/passwd"
     '(cut "-d:" "-f1")
     '(sort))
```

等价 shell：`cat /etc/passwd | cut -d: -f1 | sort`

---

### 4.3 条件分支（替代 `if` `test`）

#### 4.3.1 `run-if`

```scheme
(run-if condition-command then-command else-command)
```

`condition-command` 返回 0 时执行 `then-command`，否则执行 `else-command`。

**返回值：** `Either`
- 被执行命令成功：`Right`（内含该命令的 exit code）
- 被执行命令失败：`Left`（内含 `(list code command)`）

```scheme
(run-if '(test "-f" "Makefile")
        "make all"
        "echo 'no Makefile'")
```

---

#### 4.3.2 `run-when`

```scheme
(run-when condition-command command)
```

`condition-command` 返回非 0 时执行 `command`。

**返回值：** `Either`
- condition 返回 0（满足条件）：`Right 0`
- condition 返回非 0，command 成功：`Right`（内含 command 的 exit code）
- condition 返回非 0，command 失败：`Left`（内含 `(list code command)`）

```scheme
(run-when '(test "-e" "build/")
          "mkdir build")
```

## 5. 层级架构

```
┌─────────────────────────────────────────┐
│  基础设施层（组合逻辑）                    │
│  run-and / run-or / run-sequence         │
│  run-pipe / run-if / run-when            │
├─────────────────────────────────────────┤
│  基础原子层                               │
│  run / run-string / run-values           │
│  run-set! / run-get / run-allow!         │
│  run-ban! / run-unban!                   │
│  + keywords (:cwd :env :input :timeout ...) │
├─────────────────────────────────────────┤
│  系统调用层（隐藏实现）                    │
│  fork / execvp / pipe / dup2 / wait      │
└─────────────────────────────────────────┘
```

## 6. 与 shell 对照速查

| shell | liii_subprocess |
|-------|-----------------|
| `cmd1 && cmd2 && cmd3` | `(run-and cmd1 cmd2 cmd3)` |
| `cmd1 \|\| cmd2` | `(run-or cmd1 cmd2)` |
| `cmd1 ; cmd2 ; cmd3` | `(run-sequence cmd1 cmd2 cmd3)` |
| `cmd1 \| cmd2 \| cmd3` | `(run-pipe cmd1 cmd2 cmd3)` |
| `if test -f X; then A; else B; fi` | `(run-if '(test "-f" "X") A B)` |
| `cmd > /tmp/out 2>&1` | `(run cmd :stdout "/tmp/out" :stderr 'stdout)` |
| `cmd < input.txt` | `(run cmd :stdin "input.txt")` |
| `cmd > /dev/null` | `(run cmd :stdout 'discard)` |
| `set -e`（整个脚本） | `(run-and ...)` |

---

## 7. 设计要点

- **没有 `shell` 选项** — 想要 shell 解析就显式传 `'(sh "-c" "ls \| wc -l")`
- **env 是覆盖不是追加** — 避免 merge 逻辑歧义，和 Go 的 `cmd.Env` 一致
- **input 只接受字符串** — 不暴露流式管道接口，Scheme 里字符串更自然
- **run-string 返回 Either** — 成功为 `Right` 含 stdout，失败为 `Left` 含 exit-code，便于与 `(liii either)` 组合
- **上层设施不新增核心语义** — 全部由 run/run-string/run-values 组合而成
