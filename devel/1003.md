# [1003] 为 `gf fmt` 和 `gf fix` 增加 `--exclude` 与项目级 `gfexclude.json`

## 任务相关的代码文件
- `tools/fmt/liii/goldfmt.scm` - `gf fmt --exclude PATTERN` 参数解析、目录与变更文件遍历时按路径后缀跳过；启动时自动读取项目根的 `gfexclude.json`
- `tools/fix/liii/goldfix.scm` - `gf fix --exclude PATTERN` 同上；并把 excludes 透传给内部调用的 `gf fmt`
- `src/goldfish.hpp` - 新增 `g_project-root`，返回 cwd 下 `gfproject.json` 所在目录（找不到返回 `#f`），供 Scheme 侧定位项目根

## 如何测试
```bash
# help smoke test
bin/gf fmt --help
bin/gf fix --help

# --exclude 完整路径精确匹配（不误伤同名文件）
bin/gf fmt --exclude=a/dup.scm /tmp/mtest
# 期望：a/dup.scm 被跳过，b/dup.scm 仍被格式化

# --exclude basename 退化匹配（向后兼容）

# 多值（逗号分隔）

# 项目级 gfexclude.json 自动检测：
# 在项目根（含 gfproject.json 的目录）放 gfexclude.json，内容形如
#   {"exclude": [{"path": "a/b.scm", "reason": "..."}, "c/d.scm"]}
# 之后普通的 bin/gf fmt <dir> / bin/gf fix <dir> 会自动跳过其中文件，无需传参
bin/gf fmt /path/to/dir
bin/gf fix /path/to/dir

# 空白检查
git diff --check
```

## 2026/06/16 修复 gfexclude.json 解析被 `guard` 吞掉 + 单文件路径也尊重 exclude

### What
1. **根因**：`(scheme base)` 的 `guard` 宏在 s7 语义下，body 正常返回时**也会走 else 分支**（这是项目刻意保留的 s7 行为，`tests/scheme/base/guard-test.scm:30` 注释有说明）。原 `read-project-excludes` 把整段解析包在 `(guard (e (else <warn>)) body)` 里，body 正常返回 pattern 列表（如 `("TeXmacs/.../prefix-kbd.scm")`）后被当成"异常 e"传给 else，于是：
   - 误打 `warning: gfexclude.json 解析失败，已忽略 - (...)`
   - 实际返回 `'()`，**排除完全失效**。
2. **修法**（不动 `guard` 宏，符合 s7 语义）：抽出纯解析函数 `(parse-exclude-json text)`，`read-project-excludes` 改用 s7 原生 `catch` —— 正常返回和异常分支分开，只有真异常时才打 warning。
3. **单文件路径也尊重 exclude**：之前文档约定 `gf fmt file.scm` / `gf fix file.scm` 不应用 exclude，但与使用直觉冲突。改为：单文件分支入口加 `file-excluded?` 检查，命中时打印 `Skipped (excluded): <path>` 后正常退出。

### Why
- 用 `guard` 想做"try/catch"是这个宏最常见的误用模式；s7 的 `guard` 不是 r7rs 标准 guard，行为有差异。改用 `catch` 是更直接、更安全的写法。
- 下游（mogan）实际工作流就是 `gf fmt <某文件>`，单文件不尊重 exclude 等于配置形同虚设。

### How
- `tools/fmt/liii/goldfmt.scm`、`tools/fix/liii/goldfix.scm` 各自：
  - 新增 `(parse-exclude-json text)`：入参为已读入的 JSON 字符串，返回 pattern 列表，出错抛异常。
  - `read-project-excludes` 改用 `(catch #t (lambda () (parse-exclude-json ...)) (lambda (type info) <warn> '()))`。
  - main 的单文件分支开头加 `(if (file-excluded? path-str excludes) (begin <print skip> #t) <原逻辑>)`。

### 验证（mogan 真实场景）
```bash
# 目录路径——无 warning，被排除文件未改动
gf fmt TeXmacs/progs/texmacs/keyboard/
gf fix TeXmacs/progs/texmacs/keyboard/

# 单文件路径——打印 skip 提示，文件未改动
gf fmt TeXmacs/progs/texmacs/keyboard/prefix-kbd.scm
gf fix TeXmacs/progs/texmacs/keyboard/prefix-kbd.scm
```
四条命令下 `prefix-kbd.scm` 内容均未被修改。

## 2026/06/16 为 fmt / fix 增加 --exclude 与 gfexclude.json

### What
新增两种排除机制，互补使用：

1. **命令行 `--exclude PATTERN`**：递归遍历目录或处理 `--changed-since` 文件列表时跳过匹配的文件，多个 pattern 用逗号分隔。
2. **项目级 `gfexclude.json`**：在项目根目录（`gfproject.json` 所在目录）放一个 `gfexclude.json`，内容为 JSON 字符串数组。`gf fmt` / `gf fix` 启动时自动读取并合并进 exclude 列表，**无需传参**——普通的 `gf fmt <dir>` / `gf fix <dir>` 也会尊重它。

```bash
bin/gf fmt --exclude=TeXmacs/progs/texmacs/keyboard/prefix-kbd.scm /path/to/dir
bin/gf fix --exclude=prefix-kbd.scm /path/to/dir
bin/gf fmt --changed-since=HEAD --exclude=a.scm,b.scm
```

匹配规则（`--exclude` 和 `gfexclude.json` 条目共用，按"分隔符边界 + 后缀"，归一化路径分隔符后比较）：

- pattern 含 `/`（完整相对路径）：精确匹配。entry 等于 pattern，或 entry 以 `"/" + pattern` 结尾才命中。例如 pattern `a/dup.scm` 命中 `x/a/dup.scm` 和 `a/dup.scm`，但**不命中** `b/dup.scm`——避免重名误伤。
- pattern 不含 `/`（纯 basename）：退化为 basename 匹配，所有同名文件都命中。向后兼容简单用法。
- 归一化时同时识别 `/` 与 `\`，Windows 下反斜杠路径也能命中正斜杠写法的 pattern。
- `gf fix` 内部会对目标先调一次 `gf fmt`；这次调用同样透传 `--exclude`（用合并后的 excludes 拼成 `--exclude=a,b`），否则被排除文件仍会在 fmt 阶段被改动。
- `--exclude` / `gfexclude.json` 对单文件路径（`gf fmt file.scm`）**也生效**——见 2026/06/16 条目，原"不生效"约定已废弃。

`gfexclude.json` 示例：
```json
{
  "exclude": [
    {
      "path": "TeXmacs/progs/texmacs/keyboard/prefix-kbd.scm",
      "reason": "kbd-wildcards/kbd-map 平铺条目人工维护紧凑风格，自动格式化会重排为大段缩进+尾注释，diff 噪声大"
    },
    "other/skip.scm"
  ]
}
```

`exclude` 数组每项可以是：
- 对象 `{"path": "...", "reason": "..."}`：`path` 是要排除的文件，`reason` 说明为何排除（便于维护，reader 不校验）。
- 纯字符串：等价于只给 path，向后兼容简单写法。

### Why
此前 `gf fmt` / `gf fix` 递归遍历目录时会无差别处理所有 `.scm`，没有跳过特定文件的能力。下游项目（如 Mogan）存在一些文件受人工维护风格约束，不应被自动格式化，否则会产生与维护意图冲突的大段 diff。

设计上分两层：

- **`--exclude`** 解决"临时、单次调用"的排除需求（CI、一次性命令）。
- **`gfexclude.json`** 解决"项目长期约定"的排除需求——这些文件本就该和项目代码一起进版本库、被所有调用方一致尊重。如果只靠 `--exclude`，下游每个调用点（本地脚本、CI、开发者手动）都得记得传参，容易漏；放在项目根的 `gfexclude.json` 让"不传参的普通 `gf fmt`"也自动尊重，是更可靠的约定。

匹配规则刻意做"完整路径优先、basename 兜底"：完整路径能精确锁定单个文件、避免重名误伤（这是下游的真实诉求）；同时保留 basename 写法以兼容简单场景。

### How
#### 1. C++ 暴露 `g_project-root`

`gfproject.json` 由 C++ 侧的 `find_local_gfproject_json()` 定位（查 cwd 下）。新增 Scheme 可调函数 `g_project-root`，返回该文件的 `parent_path`（项目根目录），找不到返回 `#f`：

```cpp
static s7_pointer
f_project_root (s7_scheme* sc, s7_pointer args) {
  (void) args;
  fs::path local = find_local_gfproject_json ();
  if (local.empty ()) return s7_f (sc);
  return s7_make_string (sc, local.parent_path ().generic_string ().c_str ());
}
```

声明、名字常量、`s7_define` 三处照 `g_gfproject-load-config` 的模式添加。

#### 2. Scheme 侧 helper（goldfmt / goldfix 各自内联）

- `(normalize-sep path-str)`：把 `\` 归一化为 `/`。
- `(file-excluded? entry-str excludes)`：归一化后对每个 pattern 判断 `entry == pattern` 或 `entry` 以 `"/" + pattern` 结尾。
- `(parse-excludes raw)`：拆分 `--exclude` 逗号分隔值，丢弃空串。
- `(read-project-excludes)`：调 `(g_project-root)`，若返回字符串且项目根下存在 `gfexclude.json`，用 `(liii json)` 的 `string->json` 解析；取 `"exclude"` 字段，逐项用 `exclude-entry->path`（字符串原样返回，对象取 `"path"` 字段）转成 pattern 列表。
  - 容错：解析逻辑抽出为 `(parse-exclude-json text)`，外层用 s7 原生 `catch` 包裹（不用 `guard`——s7 的 `guard` 即使 body 正常返回也会走 else，会把返回值误当异常）。畸形 JSON 或类型不符时打 `warning: gfexclude.json 解析失败，已忽略 - ...` 并返回 `'()`，避免 `gf fmt` / `gf fix` 因配置文件写错而崩溃。
  - `"exclude"` 字段必须是 vector（`vector?` 判断），否则返回 `'()`。
  - 每个条目取出的 path 必须是字符串（`string?` 判断），否则跳过该条。

#### 3. `gf fmt` 接入

`make-fmt-arg-parser` 新增 `exclude` 参数；main 合并：

```scheme
(excludes (append (parse-excludes (parser 'exclude))
                  (read-project-excludes)))
```

`excludes` 透传给 `format-directory` / `format-file-list` / `format-changed-since`。单文件路径分支也应用 exclude（命中时打印 `Skipped (excluded): <path>`）。

#### 4. `gf fix` 接入

结构与 fmt 一致。`fix-directory` / `fix-file` 内部调 `gf fmt` 时，把合并后的 `excludes` 拼成 `--exclude=a,b` 透传：

```scheme
(let ((exclude-arg (if (null? excludes) ""
                      (string-append " --exclude=" (string-join excludes ",")))))
  (os-call (string-append (executable) " fmt" exclude-arg " " dir-path)))
```

#### 5. 帮助文档

`gf fmt --help` / `gf fix --help` 各加 `--exclude` 选项说明与一条 Example。

### Notes
- pattern 含 `/` 时按完整路径精确匹配，是排除特定文件的推荐写法；不含 `/` 时退化为 basename，仅作向后兼容。
- `gfexclude.json` 是项目级约定，应进版本库；`--exclude` 是临时/单次调用补充。两者合并生效。
- 项目根判定只查 cwd 下的 `gfproject.json`（与 `find_local_gfproject_json` 一致），不向上递归。因此 `gf fmt`/`gf fix` 应在项目根目录下运行。
- goldfish 现有测试均为子模块纯函数级，没有 CLI/目录遍历的端到端 fixture。本任务不新增单测，依赖发版后手动 smoke test 验证。
