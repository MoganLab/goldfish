## 5. 2026/06/22 njson API 统一异常捕获并按 nlohmann 异常体系分类

### 5.1 What
1. 新增 `njson_map_exception_to_error (sc, api_name, err, culprit)` 辅助函数，按 nlohmann 异常体系做分类映射：
   - `json::parse_error` → `parse-error`
   - `json::out_of_range` → `key-error`
   - `json::type_error` → `type-error`
   - 其他 `json::exception` → `value-error`
   - 其他 `std::exception` → `misc-error`
2. 新增宏 `NJSON_TRY_CATCH (sc, api_name, culprit, body)`，把 `try body catch (const std::exception&) { return njson_map_exception_to_error(...); }` 收敛到一处，避免每个函数手写两段重复 catch 子句。
3. 为以下 12 个函数补上异常捕获：
   - `njson_run_structure_conversion`
   - `f_njson_string_to_json` / `f_njson_json_to_string` / `f_njson_format_string`
   - `njson_run_value_type_predicate`
   - `f_njson_free` / `f_njson_size` / `f_njson_empty_p` / `f_njson_ref`
   - `njson_run_update`
   - `f_njson_contains_key_p` / `f_njson_keys`
   另：`njson_run_merge` 两处原本笼统归为 `type-error` 的 `catch (const std::exception&)` 也替换为 `NJSON_TRY_CATCH`，获得精细分类。
4. 只包真正可能抛 json 异常的代码段：参数校验、handle lookup、cache 维护等不抛 json 异常的代码放在 try 块外，避免捕获后破坏 `handle_store` / `handle_free_ids` / `keys_cache` 的不变量。
5. 补充测试覆盖：
   - `string-to-njson-test.scm`：+3 个非 ASCII 解析、+6 种畸形 JSON 变体（缺右括号、尾逗号、单引号、空串、纯文本、截断数组）
   - `njson-format-string-test.scm`：+1 个非 ASCII round-trip
   - `njson-to-string-test.scm`：+1 个非 ASCII round-trip

### 5.2 Why
之前 `f_njson_*` 系列里，除了 `f_njson_string_to_json` / `f_njson_format_string` 捕获 `json::parse_error`、`njson_run_merge` 笼统捕获 `std::exception` 之外，其余函数都不捕获 C++ 异常。一旦底层 nlohmann::json 抛出（深嵌套结构转换、`contains` / `operator[]` / `dump` 等异常路径），异常会冒泡到 s7 解释器，破坏 Scheme 侧的错误处理契约，Scheme 代码无法用 `(check-catch 'key-error ...)` 这类惯用法精确捕获。

之所以按 nlohmann 异常体系做映射（而不是笼统归为一个 `njson-error`），是为了让 Scheme 侧的错误处理能复用 goldfish 既有的错误符号——`parse-error` / `key-error` / `type-error` / `value-error` / `misc-error`，跟 njson API 其他入口（参数校验阶段）抛出的符号一致，调用方不必为"C++ 异常路径"和"参数校验路径"写两套捕获逻辑。

try 块范围刻意收窄到"真正会抛 json 异常的调用"：例如 `f_njson_free` 只包 `handle_store[id].reset ()`（unique_ptr 析构 json 时理论可能抛），把 `handle_free_ids.push_back (id)` 放在 try 块外——这样即使析构抛出，free-list 也不会被半更新，slot 不变量得以保留。`njson_run_update` 同理，只包 `njson_apply_update_on_root`，`njson_invalidate_keys_cache_if_present` 放外面。

### 5.3 How
- 异常分类用 `dynamic_cast` 而非多份 `catch` 子句：nlohmann 的异常全部派生自 `json::exception`（进而派生自 `std::exception`），写成多份 `catch (const json::parse_error&)` / `catch (const json::out_of_range&)` / ... 会很冗长，且顺序敏感（基类必须放后面）。单个 `catch (const std::exception&)` + `dynamic_cast` 链更简洁，分类逻辑集中在 `njson_map_exception_to_error` 一处，后续新增异常类型只改这一处。
- 宏 `NJSON_TRY_CATCH` 用 `do { try body catch (...) { return ...; } } while (0)` 形式，保证它能在需要返回 `s7_pointer` 的位置直接展开为一条语句，调用点写法跟 `return expr;` 几乎一样。
- 异常对象命名 `njson_err_`（末尾下划线）避免跟周围变量冲突。
- 没有同步 mogan 仓库在 `goldfish/liii/njson.scm` 加的 `(catch #t ...)` 包装：那层包装会把 C++ 侧精心分类的 `s7_error` 重新包装成无类型的普通 `error`，反而丢失了错误符号，是反向优化。goldfish 现有的 scm 包装函数只做参数类型预检（抛 `type-error` / `value-error`），其余直接调用 `g_njson-*`——C++ 侧抛出的结构化 error 会自然冒泡，Scheme 层 `(check-catch 'parse-error ...)` 能精确捕获。
