## 2026/04/28 使用 `(liii argparse)` 统一 tools 参数解析

### What
扩展 `(liii argparse)`，并让 `tools/` 下动态注册的 Scheme 子命令统一使用它解析参数。

本次迁移覆盖以下命令：

- `gf fmt`
- `gf fix`
- `gf test`
- `gf doc`
- `gf source`
- `gf help`
- `gf version`

需要注意的是，`argparse` 不负责决定调用哪个 tool。tool 分发仍由 C++ 主程序读取 `gfproject.json` 完成：C++ 根据子命令 import 对应 Scheme 库，然后调用该库导出的 `main`。`argparse` 只在 tool 的 `main` 被调用之后，负责解析当前 tool 的参数。

### Why
修改前，每个 tool 都在自己的入口里手写参数解析逻辑：

- `fmt` / `fix` 用 `contains?` 查找 `--help`、`--dry-run`，再用 `extract-positional` 找路径。
- `test` 自己循环跳过 `test`、`-m`、`--mode`，再把第一个剩余参数分类为 file / dir / filename / pattern。
- `doc` / `source` 自己循环跳过 `doc` / `source`、`-m`、`--mode`、`-I`、`-A`，再把剩余参数交给分类函数。
- `help` 直接读取 `(list-ref args 2)`。
- `version` 不解析参数。

这些逻辑分散在多个文件中，容易出现行为不一致，也很难复用。原有 `(liii argparse)` 又只支持“option 后面必须跟 value”的简单模式，不支持布尔 flag、位置参数、完整 `(argv)` 过滤，也不支持 `--name=value`。

因此本次目标是做一个兼容增强版本：保留已有工具的用户可见行为，同时把参数解析集中到 `(liii argparse)`。

### How
#### 1. 保留旧 API，并扩展 parser 配置

`make-argument-parser` 仍然支持原来的无参数调用：

```scheme
(define parser (make-argument-parser))
```

同时新增可选配置 alist：

```scheme
(make-argument-parser
  '((command . "fmt")
    (skip-value-options . ("-m" "--mode" "-I" "-A"))
    (skip-prefix-options . ("-m=" "--mode="))
    (unknown-options . ignore)))
```

这些配置的作用是：

- `command`：解析完整 `(argv)` 时跳过子命令名，例如 `fmt`。
- `skip-value-options`：跳过需要携带值的全局参数，例如 `-m r7rs`、`-I path`。
- `skip-prefix-options`：跳过 `--mode=liii` 这种前缀形式。
- `unknown-options`：配置未知 option 的处理方式，支持 `error`、`ignore`、`positional`。

#### 2. 新增布尔 flag 与位置参数

`add-argument` 新增 `action`：

```scheme
(parser :add-argument
  '((name . "dry-run") (action . store-true)))
```

支持：

- `store`：默认行为，option 后面读取一个 value。
- `store-true`：出现该 option 时值变成 `#t`。
- `store-false`：出现该 option 时值变成 `#f`。

解析完成后，普通参数会保存在 positionals 中：

```scheme
(parser :parse-argv '("bin/gf" "fmt" "--dry-run" "a.scm"))
(parser 'dry-run)       ; => #t
(parser :positionals)   ; => '("a.scm")
```

#### 3. 支持完整 `(argv)` 与 `--name=value`

新增 `:parse-argv`，用于接收完整命令行参数列表：

```scheme
(parser :parse-argv (argv))
```

它会跳过 executable，再根据 parser 配置跳过当前 command 和全局参数。

同时支持以下写法：

```scheme
--width=100
-w=100
```

原有写法继续保留：

```scheme
--width 100
-w 100
-width 100
```

#### 4. tools 只替换参数解析层

本次没有改变 tools 的业务逻辑，也没有改 help 文案生成方式。

例如 `gf fmt` 现在先创建 parser：

```scheme
(define (make-fmt-arg-parser)
  (let ((parser (make-argument-parser
                  '((command . "fmt")
                    (skip-value-options . ("-m" "--mode" "-I" "-A"))
                    (skip-prefix-options . ("-m=" "--mode="))
                    (unknown-options . ignore)))))
    (parser :add-argument
      '((name . "help") (short . "h") (action . store-true)))
    (parser :add-argument
      '((name . "dry-run") (action . store-true)))
    parser))
```

然后在 `main` 中解析当前 argv：

```scheme
(define (main)
  (let ((parser (make-fmt-arg-parser)))
    (parser :parse-argv (argv))
    ...))
```

`doc`、`source`、`test` 的分类规则仍保留在各自模块中，只是把“跳过 command / 全局参数 / 收集剩余参数”交给 argparse。

### Result
修改后，tools 的调用链保持不变：

1. C++ 主程序读取子命令。
2. C++ 根据 `gfproject.json` import 对应 Scheme tool。
3. C++ 调用 tool 导出的 `main`。
4. tool 在自己的 `main` 或 args 模块中使用 `(liii argparse)` 解析参数。
5. tool 根据解析结果执行原有业务逻辑。

用户可见行为保持兼容，但参数解析入口统一，后续新增 tool 或新增 option 时可以复用 `(liii argparse)`，不再需要每个工具重复写一套手工循环。

后续清理中，`gf fmt` 和 `gf fix` 原先用于手写解析的 `contains?` / `extract-positional` 已删除；对应功能现在由各自的 argparse parser 完成。
