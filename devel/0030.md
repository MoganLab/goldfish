## 2026/05/05 移除 C 层面实现，基于 char-*case 重新实现 string-*case

### What
1. 移除 `goldfish/scheme/char.scm` 中对 S7 C 内置 `string-upcase`、`string-downcase` 的依赖（删除 `s7-string-upcase`、`s7-string-downcase` 绑定）
2. 在 `goldfish/scheme/char.scm` 中导入 `(liii unicode)`，利用其 UTF-8 编解码函数实现字符串遍历
3. 新增内部辅助函数，按 UTF-8 字符粒度遍历字符串，对每个完整字符调用 `char-upcase` / `char-downcase` / `char-foldcase`
4. 重新实现 `string-upcase`、`string-downcase`、`string-foldcase`
5. 补充 UTF-8 测试用例（中文、拉丁扩展字符等）

### Why
- S7 C 内置的 `string-upcase` / `string-downcase` 基于单字节 `toupper` / `tolower`，不支持 UTF-8 多字节字符
- 已有的 `char-upcase`、`char-downcase`、`char-foldcase` 已在 Scheme 层实现了 Unicode simple casing，支持多字节字符
- 字符串版本应当与字符版本行为一致，都支持 Unicode

### How
1. **字符串遍历**：S7 的 `string-ref` 按字节索引，无法直接提取多字节 UTF-8 字符。因此借助 `(liii unicode)`：
   - `string->utf8` 将字符串转为字节向量
   - `bytevector-advance-utf8` 计算下一个 UTF-8 字符的字节边界
   - `utf8->codepoint` + `integer->char` 提取完整字符
   - 调用 `char-upcase` / `char-downcase` / `char-foldcase` 转换
   - `char->integer` + `codepoint->utf8` 转回字节向量
   - 拼接后用 `utf8->string` 得到结果
2. **移除 C 实现**：删除 `s7-string-upcase`、`s7-string-downcase` 的绑定，直接替换为纯 Scheme 实现
3. **测试补充**：在现有 ASCII 测试基础上，增加 UTF-8 字符（如中文 `中`、拉丁扩展 `é` 等）的转换验证

### 已知问题
`bin/gf fmt` 会破坏 Unicode 字符字面量。因此涉及 UTF-8 字符的测试文件不应运行 `bin/gf fmt`。
