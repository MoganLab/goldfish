## 4. 迁移策略

### 4.1 核心原则

将 `s7.c` 中的 `static s7_pointer g_xxx` 函数定义迁移到 `s7_liii_vector.c` 文件中，遵循以下模式：

1. **公开 API 优先**：使用 `s7.h` 中已有的公开 API（如 `s7_is_float_vector`、`s7_make_float_vector` 等）重写函数
2. **辅助函数桥接**：对于必须使用内部结构的场景，在 `s7_internal_helpers.h` 中暴露 `s7i_xxx` 辅助函数，在 `s7.c` 中实现
3. **保留注册信息**：`s7.c` 中的 `defun`/`bool_defun`/`s7_define_typed_function` 注册调用保持不变，仅删除 `static` 函数定义
4. **保留 H/Q 宏**：对于使用 `defun` 注册的函数，将其 `H_xxx` 和 `Q_xxx` 宏保留在 `s7.c` 中（移到文件作用域）
5. **快速路径保留**：`_p_pii`、`_p_pip`、`_p_pp` 等被内部 opt 代码直接调用的快速路径函数，保留在 `s7.c` 中不动

### 4.2 已暴露的内部辅助函数

| 辅助函数 | 用途 | 对应内部结构 |
|---------|------|------------|
| `s7i_vector_append(sc, args, typ, caller)` | vector-append 核心逻辑 | 已暴露，0035 批次 |
| `s7i_make_vector_1(sc, args, caller)` | make-vector 核心逻辑 | 已暴露，0035 批次 |
| `s7i_vector_fill_1(sc, caller, args)` | vector-fill! 核心逻辑 | 已暴露，0035 批次 |
| `s7i_vector_ref_1(sc, vect, indices)` | vector-ref 核心逻辑 | 已暴露，0035 批次 |
| `s7i_vector_ref_p_pp(sc, vec, ind)` | vector-ref 两参数快速路径 | 已暴露，0035 批次 |

### 4.3 批次计划

**第 1 批：判断函数（最简单）**
- `g_is_float_vector` → `s7_liii_vector.c`
- `g_is_int_vector` → `s7_liii_vector.c`
- `g_is_byte_vector` → `s7_liii_vector.c`
- `g_is_complex_vector` → `s7_liii_vector.c`
- 这些函数模式与 `g_is_vector` 完全一致，使用 `s7_is_float_vector` 等公开 API + `s7i_has_active_methods` + `s7i_find_method_with_let` 重写

**第 2 批：转换函数**
- `g_string_to_byte_vector` → `s7_liii_vector.c`
- `g_byte_vector_to_string` → `s7_liii_vector.c`
- 需要暴露 `s7i_make_simple_byte_vector` 或等效辅助函数

**第 3 批：简单构造函数**
- `g_float_vector` → `s7_liii_vector.c`
- `g_int_vector` → `s7_liii_vector.c`
- `g_byte_vector` → `s7_liii_vector.c`
- `g_complex_vector` → `s7_liii_vector.c`
- 需要暴露 `s7i_make_simple_float_vector` 等辅助函数，或使用 `s7_make_float_vector` 公开 API

**第 4 批：make 构造函数**
- `g_make_float_vector` → `s7_liii_vector.c`
- `g_make_int_vector` → `s7_liii_vector.c`
- `g_make_byte_vector` → `s7_liii_vector.c`
- `g_make_complex_vector` → `s7_liii_vector.c`
- 这些函数涉及多维 vector 和初始化，可能需要暴露 `make_vector_1` 等内部函数

**第 5 批：操作函数（复杂）**
- `g_vector_to_list` → `s7_liii_vector.c`（依赖 GC 保护机制，需评估）
- `g_vector_append` → `s7_liii_vector.c`（方法分派逻辑复杂，但核心逻辑 `s7i_vector_append` 已暴露）
- `g_subvector` → `s7_liii_vector.c`（依赖维度处理内部机制）

**第 6 批：设置函数**
- `g_set_vector_typer` → `s7_liii_vector.c`

**保留在 s7.c 的函数（不迁移）**

| 函数 | 保留原因 |
|------|---------|
| `vector-set!` (`g_vector_set`) | 含大量优化变体（`vector_set_p_pip`、`vector_set_p_piip` 等），fx/optimizer 和 chooser 直接引用 |
| `vector-ref` 3 参数版本 (`g_vector_ref_3`) | 被 opt 代码直接调用的快速路径 |
| `complex-vector-ref` / `float-vector-ref` / `int-vector-ref` / `byte-vector-ref` 及其 set 变体 | 被 opt 代码直接调用的快速路径 |
| `multivector` 系列 | 与求值器内部机制深度耦合 |

- 保留 `H_xxx` 和 `Q_xxx` 宏在 `s7.c` 中

### 5.2 转换函数批次（2026-05-27）

**迁移函数：**
- `g_string_to_byte_vector` → `s7_liii_vector.c`
- `g_byte_vector_to_string` → `s7_liii_vector.c`

**新增辅助函数：**
- 视情况暴露 `s7i_make_simple_byte_vector` 或相关辅助函数

### 5.3 构造函数批次（2026-05-27）

**迁移函数（简单构造函数）：**
- `g_float_vector` → `s7_liii_vector.c`
- `g_int_vector` → `s7_liii_vector.c`
- `g_byte_vector` → `s7_liii_vector.c`
- `g_complex_vector` → `s7_liii_vector.c`

**保留在 s7.c（评估后决定不迁移）：**
- `g_make_float_vector` / `g_make_int_vector` / `g_make_byte_vector` / `g_make_complex_vector`

**保留原因：**
make 构造函数涉及多维 vector 创建（`make_multivector`）、批量填充（`float_vector_fill` 等优化路径）、
rational 转 double 的特殊处理，以及大量内部错误处理宏（`error_nr`、`out_of_range_error_nr`、
`wrap_string` 等）。迁移这 4 个函数需要暴露超过 3 个新的内部辅助函数，违反评估原则，
且功能正确性风险较高，因此保留在 `s7.c`。

**新增辅助函数：**
- `s7i_make_simple_complex_vector(sc, len)` —— 供 `g_complex_vector` 使用
- `s7i_to_c_complex(z)` —— 供 `g_complex_vector` 使用

### 5.4 操作函数批次（评估后保留）

**评估结论：保留在 s7.c**

| 函数 | 保留原因 |
|------|---------|
| `g_vector_to_list` | 依赖 GC 保护机制（`check_free_heap_size`、`gc_protect_via_stack`、`begin_temp`、`return_with_end_temp`）和内部 opt 快速路径 `vector_to_list_p_p` |
| `g_vector_append` | 方法分派逻辑复杂，依赖 `copy_source_no_dest`、`make_simple_vector`、`sc->temp7` 等内部机制；`vector_append_p_pp` 快速路径被 opt 代码引用 |
| `g_subvector` | 深度依赖维度处理内部机制（`list_to_vdims`、`vdims_t`、`mallocate_empty_block`、`mark_function` 修改等），暴露成本极高 |

### 5.5 设置函数批次（评估后保留）

**评估结论：保留在 s7.c**

| 函数 | 保留原因 |
|------|---------|
| `g_set_vector_typer` | 依赖 `is_any_vector`、`is_immutable_vector`、`is_t_vector`、`global_value` 等大量内部宏和类型检查逻辑，迁移收益有限 |

## 6. 风险备忘

| 风险点 | 说明 | 缓解措施 |
|--------|------|----------|
| `static` 函数跨文件调用 | 迁出的模块需要调用 `s7.c` 中的内部辅助函数 | 将共享辅助函数移至 `s7_internal_helpers.h`，去掉 `static` 修饰 |
| 宏定义分散 | `vector_element`、`float_vector()` 等宏被 GC、Stack 多处使用 | 宏保留在 `s7.c`，只迁出函数 |
| 快速路径保留 | `_p_p`、`_p_pp` 等被 opt 直接调用 | 明确标记保留在 `s7.c`，不迁移 |
| 上游同步 | s7 upstream 是单文件库 | 在 `s7.c` 顶部保留清晰的 `/* Goldfish 改动区: ... */` 标记 |
