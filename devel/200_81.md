## s7 内存分配机制(以unicode字符为例)

### 静态对象(`< 256` 的字符) 

对于 code point 小于 `256` 的字符，s7 不会在每次读取字符时动态创建对象，而是在初始化阶段一次性创建一组静态字符对象。

在 `s7_init` 中，首次初始化时会调用 `init_chars`：

```c
if (!already_inited)
  {
    ...
    init_chars();
    ...
    already_inited = true;
  }
```

因此，在 `s7_make_character` 中：

```c
s7_pointer s7_make_character(s7_scheme *sc, uint32_t c) {
  if (c < 256) return(chars[c]);
  ...
}
```

对于 code point 小于 `256` 的字符，s7 可以直接从 `chars[]` 静态表中返回对应对象，不需要重新分配，也不会被普通 GC 回收。

### GC原理 (以 `>= 256` 的 Unicode 字符为例)

对于 code point 大于等于 `256` 的 Unicode 字符，例如：

```scheme
#\x2000
#\x3000
```

s7 无法像 `< 256` 字符那样提前全部预分配。Unicode code point 范围很大，如果全部提前分配会浪费大量内存。

因此，s7 对这类字符采用的是：

```text
按需创建 + hash table 缓存 + GC 保活
```

下面用两个对象来说明：

```text
A = unicode_chars_table 这张 hash table
B = #\x2000 对应的 T_CHARACTER 对象
```

`unicode_chars_table` 的作用是缓存已经创建过的 Unicode character：

```text
0x2000 -> B
0x3000 -> C
```

这样后续再次遇到同一个 Unicode 字符时，就可以直接从表里返回已有对象，而不是重复创建。

#### 1. 底层前提：A 和 B 都是 s7_cell 对象

s7 中普通 Scheme 对象的本体大多是一个 `s7_cell`。更准确地说：

```text
s7_cell 是对象槽位；
sc->heap 和 sc->free_heap 共同组成对象池；
new_cell 从 free_heap 中取出一个空闲 s7_cell 来创建对象。
```

`new_cell` 的核心逻辑如下：

```c
#define new_cell(Sc, Obj, Type)                         \
  do {                                                  \
    if (Sc->free_heap_top <= Sc->free_heap_trigger)     \
      try_to_call_gc(Sc);                               \
    Obj = (*(--(Sc->free_heap_top)));                   \
    set_full_type(Obj, Type);                           \
  } while (0)
```

意思是：

```text
1. 如果空闲 cell 不够，先尝试触发 GC
2. free_heap_top 往前移动一格
3. 从 free_heap 中取出一个空闲的 T_FREE cell
4. 设置对象类型
```

所以无论是：

```text
A = unicode_chars_table
B = #\x2000 character
```

它们的对象本体都是从 `new_cell` 分配出来的 `s7_cell`。

#### 2. unicode_chars_table 的创建

第一次创建 `>= 256` Unicode 字符时，会先创建 `unicode_chars_table`：

```c
if (!sc->unicode_chars_table)
  {
    sc->unicode_chars_table = s7_make_hash_table(sc, 128);
    s7_gc_protect(sc, sc->unicode_chars_table);

    if (mark_function[T_CHARACTER] == mark_noop)
      mark_function[T_CHARACTER] = just_mark;
  }
```

这里做了三件事：

```text
1. s7_make_hash_table(sc, 128) 创建 A，也就是 unicode_chars_table
2. s7_gc_protect(sc, A) 把 A 放进 protected_objects，使 A 成为 GC root
3. 将 T_CHARACTER 的 mark 函数从 mark_noop 切换成 just_mark，使 Unicode character 能被真正标记
```

其中 `s7_gc_protect` 很关键。

`sc->unicode_chars_table` 只是 `s7_scheme` 结构体里的一个 C 指针字段。GC 默认不会把所有 C 字段都当作 root 扫描，所以需要显式调用：

```c
s7_gc_protect(sc, sc->unicode_chars_table);
```

这样 GC mark 阶段会从 `protected_objects` 找到 A，从而避免 A 被回收。

`unicode_chars_table` 的组成大致如下：

```text
unicode_chars_table 的 s7_cell:
  type = T_HASH_TABLE | T_SAFE_PROCEDURE
  位于 sc->heap[] 中，由 GC 管理

unicode_chars_table 的 bucket block:
  由 callocate 分配
  hash_table_elements(A) 指向它

unicode_chars_table 的 hash entries:
  后续 s7_hash_table_set 时按需分配
```


#### 3. 创建 B：#\x2000 对应的 character

当 `s7_make_character` 遇到 `#\x2000` 时，首先会查表：

```c
s7_pointer key = s7_make_integer(sc, (s7_int)c);
s7_pointer val = s7_hash_table_ref(sc, sc->unicode_chars_table, key);

if (val != sc->F)
  return(val);
```

如果命中缓存，则直接返回已有的 `T_CHARACTER` 对象。

如果没有命中，才会创建新的 Unicode character：

```c
s7_pointer cp;
new_cell(sc, cp, T_CHARACTER | T_IMMUTABLE);
character(cp) = c;
upper_character(cp) = c;
...
```

此时 B 的结构大致是：

```text
B cell:
  type = T_CHARACTER | T_IMMUTABLE
  character(B) = 0x2000
  upper_character(B) = 0x2000
```

这里需要注意：

```text
T_CHARACTER 是主类型，表示这个对象是 character；
T_IMMUTABLE 是标志位，表示对象不可变。
```

它们不是两个“类型”，而是：

```text
主类型 + 属性标志
```

创建后，B 会被放入两个地方：

```c
add_semipermanent_object(sc, cp);
s7_hash_table_set(sc, sc->unicode_chars_table, key, cp);
```

第一处是 `unicode_chars_table`：

```text
A[0x2000] = B
```

这使得后续再次遇到 `#\x2000` 时，可以直接返回 B。

第二处是 `semipermanent_objects`：

```text
sc->semipermanent_objects -> B
```

`semipermanent_objects` 是 s7 运行时里的长期保活对象链表。它会在每次 GC mark 阶段被扫描，相当于额外的一组 GC root。

它的结构很简单：

```c
typedef struct gc_obj_t {
  s7_pointer p;
  struct gc_obj_t *nxt;
} gc_obj_t;
```

加入对象的函数是：

```c
static void add_semipermanent_object(s7_scheme *sc, s7_pointer obj)
{
  gc_obj_t *g = (gc_obj_t *)Malloc(sizeof(gc_obj_t));
  g->p = obj;
  g->nxt = sc->semipermanent_objects;
  sc->semipermanent_objects = g;
}
```

不过，仅仅加入 `semipermanent_objects` 还不够。因为它最终也是通过 `gc_mark(g->p)` 来标记对象，而 `gc_mark` 会根据对象主类型调用对应的 `mark_function`，这部分后面会讲解。

#### 4. GC 流程

GC 的本质是：

```text
从 root 出发，找到还在使用的对象并打标记；
然后遍历 heap，把没有标记的对象清理掉。
```

也就是典型的 mark-sweep：

```text
mark  阶段：给活对象打标记
sweep 阶段：删除没标记的对象
```

##### 4.1 GC 触发时机

GC 的触发入口通常在 `new_cell` 中。

当空闲 cell 不够时，会调用 `try_to_call_gc`：

```c
static void try_to_call_gc(s7_scheme *sc)
{
  if (sc->gc_off)
    {
      resize_heap(sc);
    }
  else
    {
      gc(sc);

      if ((s7_int)(sc->free_heap_top - sc->free_heap) <
          (sc->heap_size * sc->gc_resize_heap_fraction))
        resize_heap(sc);
    }
}
```

含义是：

```text
如果 GC 关闭:
  不能执行 sweep，只能扩容 heap

如果 GC 开启:
  先运行 GC
  如果 GC 后空闲 cell 仍然太少，再扩容 heap
```

##### 4.2 GC 如何找到需要保活的对象

s7 GC 有一组 root。可以把 root 理解成“联系人名单”：

```text
GC 先拿到一批确定还在使用的对象；
然后顺着这些对象引用的其他对象继续查找；
最后所有能被找到的对象都算活对象；
找不到的对象就会在 sweep 阶段被清理。
```

GC mark 阶段会从这些 root 开始：

```c
mark_rootlet(sc);                 // 全局环境里的对象
mark_owlet(sc);
gc_mark(sc->code);                // 当前正在执行的代码
gc_mark(sc->args);                // 当前函数参数
gc_mark(sc->curlet);              // 当前局部环境
mark_current_code(sc);
gc_mark(sc->value);               // 当前返回值/临时值

mark_stack_1(sc->stack, stack_top(sc)); // 运行时栈上的对象
set_mark(current_input_port(sc));
mark_input_port_stack(sc);
set_mark(current_output_port(sc));
set_mark(current_error_port(sc));
...
mark_vector(sc->protected_objects);     // 被 C 代码显式保护的对象
mark_semipermanent_objects(sc);         // 长期保留名单里的对象
```

这里与 A/B 相关的是：

```text
protected_objects -> A
semipermanent_objects -> B
A -> hash entry -> B
```

也就是说：

```text
s7_gc_protect(sc, A) 让 A 进入 protected_objects；
add_semipermanent_object(sc, B) 让 B 进入 semipermanent_objects；
A 作为 hash table，也会在 mark 阶段继续标记自己的 key/value。
```

因此，按正常的可达性模型：

```text
只要 A 存活，并且 A 的 mark 逻辑能正确标记 value，
那么 A 里面缓存的 B 也应该存活。
```

但是这次 bug 就出在“找到 B”和“真正给 B 打标记”之间。

##### 4.3 找到对象后，GC 如何标记它

找到对象后，GC 会调用：

```c
gc_mark(obj);
```

核心代码如下：

```c
static inline void gc_mark(s7_pointer p) {
  if (!is_marked(p))
    (*mark_function[type_unchecked(p)])(p);
}
```

含义是：

```text
如果这个对象还没有被标记：
  取出它的主类型
  调用该类型对应的 mark 函数
```

不同类型对象的内部结构不同，因此 mark 方式也不同：

```text
hash table:
  标记自己，还要标记 key/value

pair:
  标记自己，还要标记 car/cdr

vector:
  标记自己，还要标记每个元素

character:
  只需要标记自己
```

对于 A 来说，它的主类型是 `T_HASH_TABLE`，所以会调用 `mark_hash_table`：

```c
static void mark_hash_table(s7_pointer table)
{
  set_mark(table);
  gc_mark(hash_table_procedures(table));

  if (hash_table_entries(table) > 0)
    {
      ...
      gc_mark(hash_entry_key(entry));
      gc_mark(hash_entry_value(entry));
      ...
    }
}
```

所以：

```text
mark_hash_table(A)
  -> set_mark(A)
  -> gc_mark(entry.key)
  -> gc_mark(entry.value)，也就是 gc_mark(B)
```

对于 B 来说，它的主类型是 `T_CHARACTER`。B 创建时使用的是：

```c
new_cell(sc, cp, T_CHARACTER | T_IMMUTABLE);
```

旧代码中：

```c
mark_function[T_CHARACTER] = mark_noop;
```

而 `mark_noop` 是：

```c
static void mark_noop(s7_pointer unused_p) {}
```

也就是什么都不做。

因此修复前会出现这种情况：

```text
GC 从 root 找到了 A
A 的 mark_hash_table 又找到了 B
GC 尝试 gc_mark(B)
但是 B 的 mark_function 是 mark_noop
所以 B 没有被真正标记
```


##### 4.4 sweep 阶段如何删除对象

mark 阶段结束后，GC 会遍历整个 `sc->heap[]`：

```c
#define gc_object(Tp)                                      \
  p = (*Tp++);                                             \
  if (signed_type(p) > 0) {                                \
    clear_type(p);                                         \
    (*fp++) = p;                                           \
  } else if (signed_type(p) != 0) {                        \
    clear_mark(p);                                         \
  }
```

含义是：

```text
signed_type(p) > 0:
  说明对象有类型，但没有 T_GC_MARK
  也就是没有被 mark 阶段找到
  因此清成 T_FREE，并放回 free_heap

signed_type(p) < 0:
  说明对象带有 T_GC_MARK
  也就是本轮 GC 认为它还活着
  因此清掉 mark bit，保留对象

signed_type(p) == 0:
  说明本来就是 T_FREE
```


##### 4.5 这次段错误的形成过程

修复前的流程如下：

```text
1. 第一次遇到 #\x2000
2. 创建 A = unicode_chars_table，并通过 s7_gc_protect 保护 A
3. 创建 B = #\x2000 对应的 T_CHARACTER 对象
4. 将 B 存入 A：A[0x2000] = B
5. GC 触发
6. GC 从 protected_objects 找到 A
7. mark_hash_table(A) 遍历 entry，找到 B
8. GC 调用 gc_mark(B)
9. 因为 mark_function[T_CHARACTER] = mark_noop，B 没有被设置 T_GC_MARK
10. sweep 阶段发现 B 没有 mark，于是将 B 清成 T_FREE
11. 但是 A 仍然存活，A 的 entry.value 仍然指向 B 的旧地址
12. 后续再次访问 #\x2000 时，s7_hash_table_ref 从 A 中返回这个旧地址
13. 旧地址可能已经是 T_FREE，或者已经被复用成别的对象，最终触发段错误
```

所以问题的本质是：

```text
A 活着，但 A 缓存的 B 被 GC 提前清理了。
```

##### 4.6 修复后的流程

修复后有两个关键变化：

```c
if (mark_function[T_CHARACTER] == mark_noop)
  mark_function[T_CHARACTER] = just_mark;
```

以及：

```c
add_semipermanent_object(sc, cp);
```

于是 B 有两条保活路径：

```text
protected_objects -> A -> hash entry -> B
semipermanent_objects -> B
```

并且 B 的 mark 函数变成了 `just_mark`：

```text
gc_mark(B)
  -> mark_function[T_CHARACTER](B)
  -> just_mark(B)
  -> set_mark(B)
```

因此 sweep 阶段看到 B 时：

```text
signed_type(B) < 0
```

说明 B 已经被标记为活对象，GC 只会清掉它的 mark bit，不会把它清成 `T_FREE`。

最终结果是：

```text
A 存活
B 存活
A[0x2000] 仍然指向有效的 B
再次访问 #\x2000 时不会返回悬空指针
```

#### 总结：Mark-Sweep 在 S7 中的简化版流程
对象分配 (new_cell)
    │
    ▼
free_heap 是否充足？ ──否──▶ try_to_call_gc()
    │                           │
    是                          ▼
    │                       gc() 启动
    ▼                           │
返回 cell                  ┌────┴────┐
                           ▼         ▼
                        Mark      Sweep
                           │         │
                           ▼         ▼
                   从 Roots 递归    遍历 heap[]
                   给活对象打标记   未标记 → free_heap
                           │         │
                           └────┬────┘
                                ▼
                         free_heap 是否仍不足？
                              │
                    ┌───────否───────┐
                    ▼                ▼
              返回 new_cell    resize_heap()
                                    │
                                    ▼
                              heap_size *= 2
                              分配新 cell
                              加入 free_heap

## 2026/04/29 修复 gf fmt goldfish/liii/ 的段错误
### What
修复 `bin/gf fmt goldfish/liii/` 在导入 formatter 过程中触发的段错误。

1. 定位 `gf fmt` 崩溃路径，确认问题发生在 `src/s7.c` 的字符对象 GC / remove-from-heap 逻辑。
2. 修复 Unicode `character` 对象被重复移除或被 GC 提前释放的问题。
3. 验证 `gf fmt` 递归格式化目录时不再因为 `s7.c` 中的字符对象生命周期错误而崩溃。

### Why
`gf fmt goldfish/liii/` 是一个直接可复现的崩溃问题，会导致格式化工具无法用于整个目录。

进一步定位后发现，`s7_make_character` 为大于 `255` 的 Unicode 字符创建了可共享的字符对象，但这些对象在闭包优化和 GC 过程中没有被当作长期存活对象正确处理，最终会出现悬空指针或重复 remove-from-heap，进而触发段错误。

### How
在 `src/s7.c` 中做了三处修复：

1. 在 `remove_from_heap` 中对 `character` 对象直接跳过，并对已 free 的 cell 先行返回，避免共享字符进入错误的 remove-from-heap 路径。
2. 在初始化 `unicode_chars_table` 时，将 `T_CHARACTER` 的 mark 函数从 `mark_noop` 切换为 `just_mark`，确保缓存的 Unicode 字符在 GC 标记阶段会被真正标记。
3. 在创建 Unicode `character` 对象后，将其加入 `semipermanent_objects`，使其与缓存表的共享语义一致，避免被 GC 提前回收。

## 如何测试

```bash

xmake build goldfish
```

2. 直接验证原始问题不再复现：
```bash

```

3. 为了避免修改工作区源码，也可以在临时目录验证：


预期结果：
- 命令正常完成。
- 不出现 segmentation fault / SIGSEGV / attempt to use free cell。
- 结束时输出 `Total files processed: 53` 一类的统计信息。
