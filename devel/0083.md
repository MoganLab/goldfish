## 2026/06/28 string->utf8 下沉到 native C++
### What
将 `string->utf8` 从纯 Scheme 实现下沉到 native C++（放在新建的 `src/scheme_base.cpp`，归属 `(scheme base)`），完全保留 [前述测试整合](#20260628-为-stringutf8-新增完整的测试用例) 固化的 47 个 check 语义契约，同时获得数百到数千倍的性能提升。

1. 新增 `src/scheme_base.cpp`：实现 `f_string_to_utf8`，以 `g_string->utf8` 注册到 s7；用 `s7_string` + 公开 API `s7_string_length` 拿到含 NUL 的字节缓冲，自己实现 UTF-8 字符宽度判定与字符定位
2. 在 `src/goldfish.hpp` 增加 `glue_scheme_base` 声明，并在 `glue_for_community_edition` 中调用
3. 在 `xmake.lua` 的 goldfish target 中加入 `src/scheme_base.cpp`
4. 改写 `goldfish/scheme/base.scm`：`string->utf8` 直接转发到 `g_string->utf8`，删除原 Scheme 的 `string->utf8-sub` 循环及 `bytevector-advance-utf8` 依赖
5. 新增 `bench/string-to-utf8-perf.scm`：覆盖 1/2/3/4 字节码点 × 5 个长度档位 + start/end 切片，字符串预构造避免构造开销污染测量

### Why
`string->utf8` 是 `(liii base64)` 的 `string-base64-encode`、JSON、HTTP 等高频路径的底层依赖。Scheme 实现的主循环 `string->utf8-sub` 每个字符都要调用 `bytevector-advance-utf8` 做 Scheme 层字节判定，固定开销大（ASCII 长度 1 单次就 ~22µs），且随码点宽度线性恶化。下沉到 native 后，`(scheme base)` 的 string↔bytevector 转换两端（`string->utf8` 与之前已优化的 `utf8->string`）均跑在 native 层。

### How
- 主流程：`s7_string_length` 取字节长度 → 一次遍历统计字符总数 N 并校验 UTF-8 合法性 → 按 start/end 校验区间 → 再一次遍历定位 start/end 对应的字节偏移 → `memcpy` 切片到新 bytevector
- UTF-8 字符宽度用首字节 leading-zeros 模式判定（`<0x80`→1, `0xE0`掩码→2, `0xF0`→3, `0xF8`→4），不调用 s7 内部宏
- 参数解析注册为 rest 函数 `(1, 0, true)`，手动遍历 args 列表；`end` 缺省或传 `#t` 均按「取到末尾」处理（与 Scheme 端 `define*` 默认值对齐）
- 错误契约与 Scheme 实现逐条对齐：非 string → `type-error`；start/end 越界、start==N、start>end → `out-of-range`；start==end → 空 bytevector；非法 UTF-8 → `value-error`
- 含 NUL 的字符串：`strlen` 会被嵌入 NUL 截断，改用公开 API `s7_string_length`（注释即 `(string-length str)`）拿真实字节长度

### 性能对比（debug 构建，单次平均耗时，字符串预构造）
| 长度 | ASCII(1B) before → after | 加速 | 中文(3B) before → after | 加速 |
|---|---|---|---|---|
| 1 | ~22µs → ~2.1µs | ~11x | ~29µs → ~2.2µs | ~13x |
| 10 | ~133µs → ~2.2µs | ~60x | ~198µs → ~2.3µs | ~87x |
| 100 | ~1.25ms → ~2.8µs | ~446x | ~1.88ms → ~3.1µs | ~606x |
| 1000 | ~12.3ms → ~10µs | ~1230x | ~18.7ms → ~13µs | ~1438x |
| 10000 | ~123ms → ~82µs | ~1494x | ~193ms → ~97µs | ~1990x |

start/end 切片（1000 字符中文字符串）：全程 ~19ms→~14µs (~1357x)、前半 ~14ms→~12µs (~1167x)、后半 ~20ms→~13µs (~1538x)、中段 ~17ms→~13µs (~1365x)。

整轮 bench 耗时从约 40s 降至约 1.1s。Emoji(4B) 与拉丁(2B) 表现同中文档位，码点宽度对 native 实现几乎无影响（仅循环次数与 memcpy 长度差异）。

## 2026/06/28 为 string->utf8 新增完整的测试用例
### What
将 `string->utf8` 原本分散在两处、且各有缺口的测试整合为一份完整覆盖的测试文件 `tests/scheme/base/string-to-utf8-test.scm`，共 47 个 check。

1. 删除 `tests/liii/unicode/string-to-utf8-test.scm`（与 base 版重叠，且编码正确性用例并入新文件）
2. 重写 `tests/scheme/base/string-to-utf8-test.scm`，按五个维度组织：空串/ASCII、全码点范围（1~4 字节）、start/end 参数边界、错误契约、round-trip 互逆
3. 补齐原两份测试各自的缺口：start==N、start>end、start==end（非 0）、end==N、空串+start、多字节字符级切片、参数缺失/类型错误（非 string、整数、symbol）、负值越界

### Why
`string->utf8` 属于 `(scheme base)`，此前却同时存在 `tests/scheme/base/string-to-utf8-test.scm`（9 check，侧重接口契约）和 `tests/liii/unicode/string-to-utf8-test.scm`（19 check，侧重编码正确性）两份测试，覆盖面重叠但任何一份单独都不完整——例如 base 版缺 4 字节 emoji 与 start==end(非0)，unicode 版缺参数缺失、类型错误、start 负值。整合到 base 一份既符合函数归属，也消除了维护两份的漂移风险，并为后续可能下沉 native 的回归提供完整基线。

### How
- 先用临时探针验证不确定的边界契约（start==N、start>end、start==end、end==N、空串+start），确认实际行为后再落正式用例，避免测错契约
- NUL(`\x00;`) 与 DEL(`\x7f;`) 用 `(string (integer->char n))` 构造，避免源码字面量被 `gf fmt` 渲染成实际控制字符而丢失可读性
- 错误契约覆盖三类：`type-error`（参数缺失 / 非 string）、`out-of-range`（start/end 越界、start==N、start>end）
- round-trip 维度复用已下沉 native 的 `utf8->string`，两端互为交叉校验，含多字节与 start/end 切片
