## 2026-06-25 clang-format binary 配置支持数组

### What
1. `lang-binary`（`goldfmt-config.scm`）的 `binary` / `binary-linux` / `binary-windows` / `binary-macos` 字段从仅接受字符串，扩展为同时接受字符串数组：数组按顺序取第一个存在且可执行的路径。
2. 把 binary 解析逻辑拆成三个职责单一的纯函数：`binary-usable?`（单候选是否可用）、`binary-candidates`（JSON 值展开为候选列表）、`first-usable-binary`（列表里取第一个可用），`lang-binary` 复用同一查找逻辑。
3. `gf_fmt.json` 的 `cpp.binary-linux` 改为 `["/usr/bin/clang-format", "/usr/bin/clang-format-19"]`，覆盖 Debian 13 CI 安装 `clang-format-19` 的实际路径；新增 `cpp.binary-macos` 为 `["/opt/homebrew/opt/llvm/bin/clang-format", "/usr/local/opt/llvm/bin/clang-format"]`，覆盖 Apple Silicon 与 Intel 两种 Homebrew llvm 安装位，使本地 macOS 的 `gf fmt` 能找到 clang-format。
4. 新增 `tools/fmt/tests/liii/goldfmt-config-test.scm`，覆盖字符串配置、数组首项不存在落到次项、全不存在回退默认、裸命令名视为可用、os 字段优先级、suffix 补点/丢弃空串/回退默认等用例。
5. 用标准库函数简化 `goldfmt-config.scm`：`read-string-array` 改用 `(filter string? (vector->list ...))`、`parse-exclude-array` 改用 `(filter identity (map ...))`、`first-usable-binary` 改用 `find`，并抽取 `valid-root?` 消除 `config-exists?` / `load-fmt-config` 重复的 root 校验。
6. 修复 `normalize-suffix` 的空串 bug：原实现对空串会返回 `"."`（误匹配一切），改为返回 `#f` 并由 `lang-suffixes` 过滤丢弃，与注释「丢弃空串」一致。

### Why
CI（`ci-format-check-debian13.yml`）通过 `apt-get install clang-format-19` 安装，实际二进制是 `/usr/bin/clang-format-19`，并不存在 `/usr/bin/clang-format`。原先 `binary-linux` 只能配单个字符串，配 `/usr/bin/clang-format` 时该文件不存在，`clang-format --version` 子进程调用失败，导致 CI 始终报「未找到 clang-format」并误判文件需格式化。改为数组后，可在配置中按优先级列出多个候选路径，自动落到实际存在的那一个。

### How
- 候选路径是否「可用」的判定：含路径分隔符（`/` 或 `\`）的视为显式路径，用 `file-exists?` + `(liii os)` 的 `access X_OK` 检查存在性与可执行位；裸命令名（如 `clang-format`）视为 PATH 查找，恒可用，交给子进程解析。
- 优先级：当前平台 os 字段（`binary-linux` 等）整体高于通用 `binary`；任一字段为字符串或数组都按上述规则取第一个可用者；都不可用时回退默认 `"clang-format"`。

## 2026-06-24 gf fmt 为 clang-format 添加文件缓存

### What
1. 把 `scheme-fmt.scm` 中的缓存函数抽取到新的共享模块 `goldfmt-cache.scm`，供 Scheme 与 C++ 格式化复用。
2. 在 `cpp-fmt.scm` 的 `format-cpp-file`、`format-one-cpp`、`check-cpp-file` 中增加缓存命中/写入逻辑：
   - 命中缓存时直接跳过 `clang-format` 调用；
   - 未命中时调用 `clang-format -i` 原地格式化，通过比较格式化前后内容判断是否有变更；
   - `--check` 模式下同样使用缓存，检查通过后写入缓存。
3. `format-cpp-files` 返回 `(total updated cached)`，与 Scheme 格式化协议保持一致。
4. 新增 `tests/liii/goldfmt-cache-test.scm`，覆盖缓存路径计算、命中/写入逻辑。
5. 在 `(liii path)` 中新增 `path-starts-with?` 函数，用于跨平台判断路径是否为另一路径的子路径，并在 `goldfmt-cache-test.scm` 中用它替代字符串比较。
6. 使用 `(liii subprocess)` 的 `run` 重构 `clang-format` 调用，移除 `sh -c` 包装与 shell 字符串拼接。
7. `clang-format-binary` 改为优先读取 `gf_fmt.json` 中的 `cpp.binary` / `cpp.binary-linux` / `cpp.binary-windows` / `cpp.binary-macos`，未配置时回退到 PATH 中的 `clang-format`；并在当前仓库配置 `binary-linux` 为 `/usr/bin/clang-format`。未找到时给出提示。
8. 修复目录模式（`gf fmt -e cpp .`）未加载 `gf_fmt.json` 的问题：存在配置时以配置中的路径、后缀、排除项及二进制路径为准。

### Why
`gf fmt` 在格式化大量 C++ 文件时，每次都要调用 `clang-format` 进程，即使文件内容没有变化也无法跳过。`clang-format` 本身没有持久化缓存选项，因此在反复格式化同一个仓库时浪费大量时间。通过文件内容 SHA-256 缓存，可以把第二次格式化的耗时从数十秒降到亚秒级。

### How
缓存方案与 Scheme 格式化现有缓存一致：
- 缓存目录：`~/.cache/goldfish/fmt/<version>/`
- 缓存键：源文件内容的 SHA-256
- 缓存文件路径采用两级目录 `<前2位hash>/<剩余62位hash>`
- 缓存失效：Goldfish 版本变更会自动切换缓存目录；`.clang-format` 等配置变更需要手动清理缓存（与 Scheme 缓存语义一致）

在 `~/git/mogan` 上实测：
- 首次格式化：约 36 秒
- 再次格式化：约 0.3 秒
