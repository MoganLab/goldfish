## 2026-03-30 拆分 either-test.scm

### What
将 `tests/goldfish/liii/either-test.scm` 中聚合在一个文件里的测试，按函数拆分到 `tests/liii/either/` 目录中，并新增轻量入口文件 `tests/liii/either-test.scm`。

本次迁移覆盖的函数包括：

1. **构造与提取**
   - `from-left` -> `from-left-test.scm`
   - `from-right` -> `from-right-test.scm`
   - `to-left` -> `to-left-test.scm`
   - `to-right` -> `to-right-test.scm`

2. **谓词**
   - `either-left?` -> `either-left-p-test.scm`
   - `either-right?` -> `either-right-p-test.scm`

3. **高阶操作**
   - `either-map` -> `either-map-test.scm`
   - `either-for-each` -> `either-for-each-test.scm`

4. **实用函数**
   - `either-get-or-else` -> `either-get-or-else-test.scm`
   - `either-or-else` -> `either-or-else-test.scm`

5. **逻辑判断与过滤**
   - `either-filter-or-else` -> `either-filter-or-else-test.scm`
   - `either-contains` -> `either-contains-test.scm`
   - `either-every` -> `either-every-test.scm`
   - `either-any` -> `either-any-test.scm`

### 注释格式
每个拆分后的测试文件都采用统一的文档注释格式：

```scheme
;; 函数名

;; ----

;;
;; 错误处理
;; ----
;; 异常类型与触发条件
```

### 文件命名规则
本次迁移沿用统一的特殊符号转写规则：

1. `?` -> `-p`
2. `!` -> `-bang`
3. `->` -> `-to-`
4. `=` -> `-eq`
5. `*` -> `-star`
6. `/` -> `-slash-`
7. 关系运算符中的 `<` / `>` / `<=` / `>=` 按语义写法转写

### Why
1. `tests/goldfish/liii/either-test.scm` 聚合了多个独立的 Either API 测试，单文件维护成本偏高。
2. 拆分为单函数测试文件后，便于单独执行、回归和审查。
3. 与最近的 `string` / `vector` / `ascii` / `base` 迁移方式保持一致，目录结构更统一。
4. 为每个测试文件补齐统一格式的文档注释，便于后续维护。

### How
1. 从原 `either-test.scm` 中按函数边界提取测试代码。
2. 将综合流程和异常测试并入最相关的函数文件，同时保证单函数测试文件只断言对应目标函数的行为。
3. 按“一个函数一个文件”的方式迁移到 `tests/liii/either/`。
4. 为每个新文件补齐统一的文档块注释。
5. 删除旧的聚合测试文件 `tests/goldfish/liii/either-test.scm`。
6. 新增 `tests/liii/either-test.scm`，仅用于说明测试已迁移到目录中。
7. 使用项目真实测试入口 `bin/goldfish -m r7rs` 逐个执行 `tests/liii/either/` 下的所有测试文件，确认迁移后全部通过。
