## 2026/05/05 为 hash-table 初始化添加 defined? 保护

### What
为 `*char-numeric-ht*`、`*char-upper-ht*`、`*char-lower-ht*` 三个 hash table 的初始化代码添加 `unless (defined? '...)` 保护，避免多次导入 `(scheme char)` 时重复执行大量 `hash-table-set!` 操作。

1. 在 `goldfish/scheme/char.scm` 中，三个 hash table 的 `define` 和填充代码均用 `unless (defined? '...)` 包裹
2. 添加 `tests/liii/base/defined-p-test.scm` 测试用例，验证 `define` 在 `unless` 内部的行为

### Why
`(scheme char)` 库被多次加载时（如 REPL 中重复导入），三个 hash table 的 2000+ 条 `hash-table-set!` 调用会被重复执行，造成不必要的性能开销。

### How
利用 S7 的 `defined?` 函数检查符号是否已在当前环境中绑定。若已定义则跳过整个初始化块，否则执行 `define` 和填充逻辑。测试验证 `define` 在 `unless` 内部可正常工作且变量作用域正确。

## 2026/05/05 将字符分类函数改为 hash-table 实现

### What
将 `char-numeric?`、`char-upper-case?`、`char-lower-case?` 从基于 `cond` 的区间判断改为基于 s7 内置 hash-table 的实现，并添加性能测试。

1. 在 `goldfish/scheme/char.scm` 中定义统一的辅助函数 `char-ht-set-range!`
2. `char-numeric?` 使用 hard code 的 152 个区间初始化 hash-table（共 1919 个字符）
3. `char-upper-case?` 使用 hard code 的 651 个区间初始化 hash-table
4. `char-lower-case?` 使用 hard code 的 675 个区间初始化 hash-table
5. 三个函数均改为 `(hash-table-ref *ht* (char->integer char))` 查询
6. 创建 `bench/char-numeric-p.scm`、`bench/char-upper-case-p.scm`、`bench/char-lower-case-p.scm` 性能测试文件

### Why
原实现基于 `cond` 的区间判断，对于不匹配任何区间的字符（如 ASCII 字母调用 `char-numeric?`），需要遍历所有分支才能到达 `else #f`，导致严重的性能退化。以 `char-numeric?` 为例，ASCII 字母场景比 ASCII 数字场景慢约 20 倍。

### How
采用 s7 内置的 `s7-make-hash-table`、`s7-hash-table-set!`、`hash-table-ref` 构建查询表。hash-table 的命中/不命中查询均为 O(1)，消除了 cond 遍历的性能瓶颈。初始化采用 hard code 区间的方式，在模块加载时一次性完成，避免运行时扫描 Unicode 码点的开销。

### 性能对比（10 万次迭代）

| 测试用例 | 旧实现 (cond) | 新实现 (hash-table) | ascii 版 |
|---------|--------------|--------------------|----------|
| char-numeric? #\5 | 0.0047 秒 | 0.0041 秒 | 0.0040 秒 |
| char-numeric? #\a | 0.0967 秒 | 0.0038 秒 | 0.0037 秒 |
| char-upper-case? #\A | ~0.09 秒 | 0.0040 秒 | 0.0037 秒 |
| char-upper-case? #\a | ~0.09 秒 | 0.0043 秒 | 0.0038 秒 |
| char-lower-case? #\a | ~0.09 秒 | 0.0040 秒 | 0.0036 秒 |
| char-lower-case? #\A | ~0.09 秒 | 0.0039 秒 | 0.0037 秒 |
