## 3. gf fmt 用法

```bash
gf fmt                       # 仓库批量格式化所有语言（读 gf_fmt.json：C++ + Scheme）
gf fmt --check               # CI 非破坏检查，未格式化则退出码 1
gf fmt file.scm              # 格式化单个文件（按后缀选语言）
gf fmt a.cpp                 # cpp 单文件转 clang-format（比对内容，诚实统计）
gf fmt --dry-run file.scm    # 预览格式化结果（不写回）
gf fmt dir/                  # 递归格式化目录下所有 .scm 文件（默认 scheme）
gf fmt -e cpp src/           # 目录模式选 cpp 语言，涵盖 .cpp/.hpp/.h/.c/.cc/.cxx
gf fmt -e scm,sld dir/       # 也可直接写后缀（逗号分隔多个，须同语言）
gf fmt --changed-since=HEAD  # 仅格式化自 HEAD 以来变更的 Scheme 文件
gf fmt --exclude=PATTERN ... # 追加排除（逗号分隔多个）
```

`-e` 接受语言名（`cpp`/`scheme`，自动涵盖该语言全部后缀）或具体后缀（`scm`/`sld`）。目录模式要求 `-e` 的后缀同属一种语言，混合（如 `-e scm,cpp`）报错。

入口分发顺序（`cond` 自上而下短路）：
- `--changed-since` 优先：用户传了该 flag 时（无论是否带路径）一律走 Scheme 增量。
- 无路径参数：读 `gf_fmt.json`，`--check` 走非破坏检查，否则仓库批量；配置缺失则报错退出 1。
- 有路径参数：单文件按文件后缀查注册表选语言 handler；目录按 `-e` 后缀选语言 handler（cpp 转 clang-format，scheme 走 goldfmt 引擎）。

## 4. gf_fmt.json 格式

放项目根（`g_project-root`，即 `gfproject.json` 所在目录）：

```json
{
  "cpp": {
    "suffix": "hpp,cpp,h,c,cc,cxx",
    "path": "src",
    "exclude": [
      {"path": "src/s7*", "reason": "s7 体系文件，沿用 s7 自身代码风格"},
      "tests"
    ]
  },
  "scheme": {
    "suffix": "scm",
    "path": "goldfish,tools",
    "exclude": ["tests"]
  }
}
```

- `suffix`：逗号分隔的后缀（不带点）；未写则用语言默认（cpp=`hpp,cpp,h,c,cc,cxx`，scheme=`scm`）。
- `path`：逗号分隔的根目录，仓库批量/check 时递归收集。
- `exclude`：**每语言各自**。同时支持两种格式——纯字符串 `"src/s7*"` 或对象 `{"path":"...","reason":"..."}`。支持通配符 `*`。

排除匹配规则（`goldfmt-lang.scm` 的 `path-matches-exclude?`）：
- pattern 含 `*`：glob 匹配。含 `/` 时对完整路径匹配（`src/s7*` 精确到 src 下），否则对 basename 匹配（`test*` 匹配任意目录下 test 开头）。
- pattern 不含 `*`：精确路径匹配（等于 pattern，或以 `/pattern` 结尾，使 `tests` 同时匹配 `tools/fmt/tests` 与 `tests`）。

## 7. 设计要点

### 7.1 What
1. `gf fmt` 一个命令覆盖全场景：单文件、目录递归、仓库批量、CI 非破坏检查（`--check`）、增量（`--changed-since`）。
2. 单一配置文件 `gf_fmt.json` 按语言（`cpp` / `scheme`）组织，承载扫描范围（`path`）与排除（`exclude`）。
3. 按语言拆分实现：`goldfmt-config`（配置）、`goldfmt-lang`（语言注册表 + 通用工具）、`scheme-fmt`、`cpp-fmt`，主入口 `goldfmt.scm` 仅做参数解析与按注册表分发。每个语言模块自己声明接管的后缀表（`cpp-extensions`/`scheme-extensions`）并注册 handler。**新增语言零改主入口**：写一个 `xxx-fmt.scm`（声明后缀表 + 实现协议 + 注册）+ import 一行 + `gf_fmt.json` 增一段即可，全场景自动覆盖。
4. Debian 13 格式化检查 CI：构建 `bin/gf` 后运行 `gf fmt --check`。

### 7.2 Why
goldfish 已有 `gf fmt`（Scheme）和 `.clang-format`（C++），需要**统一入口**和 **CI 级格式化门禁**：C++ 此前无自动化格式化/检查，Scheme 只能逐目录手动跑。把仓库批量与 CI 检查合并进 `gf fmt`、配置集中到 `gf_fmt.json`，使命令职责清晰、配置不再割裂、跨语言可扩展。

扫描范围配置化而非硬编码，是为了让**同一个 `gf fmt` 在不同仓库复用**：下游仓库只需放一份自己的 `gf_fmt.json` 即可。

排除上需要特殊对待两类文件：
- **s7 体系文件**（`src/s7*`）：clang-format 会破坏其脆弱的宏定义与排版（实测导致 `use_write_t`/`shared_info_t`/`sigjmp_buf` 等编译错误），必须整体排除。
- **测试文件**（`tests`）：含大量故意畸形的夹具（缺括号等，交给 fmt 会抛 read-error），且改动频繁，纳入仓库级 format 既慢又易误报；测试格式化由各测试自身负责。

### 7.3 How
- **语言注册表真派发**：`(liii goldfmt-lang)` 维护语言→handler 注册表（`register-lang!`，按模块 import 顺序追加）。各语言 handler 是一个关联列表，统一协议：
  - `name`（符号）、`label`（人类可读名，用于日志）、`extensions`（该语言接管的后缀表，带点，如 cpp 的 `(".cpp" ".hpp" ".h" ".c" ".cc" ".cxx")`、scheme 的 `(".scm")`——后缀表由各语言模块自己声明）
  - 仓库批量/check 方法（接 `cfg`，handler 内部用 `goldfmt-config` 的 `lang-suffixes`/`lang-paths`/`lang-excludes` 自取本语言配置）：
    `(collect cfg)`、`(format-files files cfg)` → 返回 `(total updated unchanged)`、`(check-file path cfg)` → `#t`/`#f`
  - 带路径参数方法（接调用方传入的后缀/excludes）：`(format-file path dry-run excludes)`、`(format-directory dir exts excludes dry-run)` → 返回 `(total updated unchanged)`
- **主入口零语言知识**：仓库批量/check 遍历 `(lang-list)` 用 `lang-ref` 取方法派发；单文件按文件后缀 `lang-for-extension` 选 handler、目录按 `-e` 后缀 `lang-for-extensions` 选 handler。**主入口无任何 `(eq? lang 'scheme/'cpp)` 硬编码分支**。新增一门语言只需写一个 `xxx-fmt.scm`（声明 extensions + 实现协议 + `register-lang!`）并在 `goldfmt.scm` 的 import 里加一行、`gf_fmt.json` 加一段，全场景自动覆盖。
- **`-e` 按语言名展开**：`-e cpp` 自动涵盖 cpp 的全部后缀（`.cpp/.hpp/.h/.c/.cc/.cxx`），`-e scheme` 涵盖 `.scm`；也可直接写后缀 `-e scm,sld`（`extensions-for-lang-name` 查注册表，是语言名则展开为该语言后缀表，否则按单后缀处理）。
- **目录模式语言一致性**：`-e` 指定的后缀必须全部属于同一语言（`-e hpp,cpp` 同属 cpp OK），混合多语言（`-e scm,cpp`）报错退出——避免一个目录被多个引擎同时处理造成混乱。
- **退出码**：s7 tool 的 `main` 返回值由 C++（`goldfish_finish_tool_success`）转为进程退出码；`(exit N)` 也会真正终止进程并传码。故 check 失败、配置缺失、目录混合后缀等路径都用 `(exit N)` 传码。
- **exclude 复用**：`glob-match?` / `path-matches-exclude?` / `file-excluded?` / `collect-files` 作为通用层放在 `goldfmt-lang.scm`，scheme-fmt 与 cpp-fmt 共用。
- **check 实现**：Scheme 逐文件 `scan-file`+`format-nodes` 与磁盘逐字节比；C++ 逐文件 `clang-format --dry-run --Werror` 看退出码（Windows 无 sh 跳过，CI 在 Debian 跑）。check 结果汇总遍历求和（不按固定语言 key 取，避免新语言漏算）。
- **格式化统计诚实**：**cpp 不做缓存**（与 scheme 的 sha256 文件缓存不同）。cpp 单文件/批量格式化时，把 `clang-format` 结果输出到临时文件（用 `sh -c` 包裹重定向，因 `os-call` 不走 shell）与原内容逐字节比对——不同才写回（计 `updated`），相同跳过（计 `unchanged`）。故 `updated` 只数真实改动，不会假报。统计行统一为 `Total X, Files updated: Y, Files unchanged: Z`。
- **不重写算法**：Scheme 格式化核心（goldfmt-scan/format）、文件缓存、clang-format 定位均沿用既有实现。

### 7.4 工具定位
`gf fmt` **本质是 Scheme 格式化工具**（核心引擎为 `goldfmt-scan`/`goldfmt-format`）。C++ 等其他语言是通过在注册表里注册一个 handler、内部转调外部工具（clang-format）来支持的。带路径参数时按后缀/语言名选语言：
- 默认按 Scheme 处理（`-e` 默认 `scm`）。
- 单文件：cpp 后缀（`.cpp`/`.hpp`/...）转 clang-format（比对内容、诚实统计），其余走 scheme 引擎。
- 目录：`-e cpp`（或 `-e hpp,cpp`）收集目录下 cpp 后缀文件交 clang-format 逐文件处理；`-e scm` 走 scheme 引擎递归；混合语言后缀报错。**注意：cpp 文件绝不进 scheme 引擎**（否则会被当 scheme 代码解析、破坏性乱改）。
- 无路径参数（仓库批量/check）才按 `gf_fmt.json` 把所有已注册语言（含 C++ 经 clang-format）一并处理。

> 历史坑：早期目录分支只走 scheme 的 `format-directory`，导致 `gf fmt -e cpp dir/` 用 scheme 引擎破坏 cpp 文件。现按后缀在主入口分流，cpp 单文件/目录统一转 clang-format；cpp 统计也从"`-i` 一把梭假报 updated"改为逐文件比对内容的诚实统计。
