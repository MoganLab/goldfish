## 3. 如何测试

### 3.1 确定性测试
```bash
xmake b goldfish

# 子命令已注册
bin/gf help | grep format

# 非破坏检查：仓库已全部格式化时应输出 OK、退出码 0
bin/gf format --check; echo "exit=$?"

# 原地批量格式化
bin/gf format
```

### 3.2 非确定性测试（行为验证）
```bash
# 人为制造一处未格式化的 C++ 文件，确认 --check 能报告并退出码 1
cp src/goldfish.cpp /tmp/goldfish.cpp.bak
printf '\n\n\nint x;\n' >> src/goldfish.cpp
bin/gf format --check; echo "exit=$?"   # 期望 exit=1，并列出 src/goldfish.cpp
cp /tmp/goldfish.cpp.bak src/goldfish.cpp   # 还原

# 排除验证：s7.h 被 gfexclude.json 排除，制造异常不应被报告
printf '\n\n\nint x;\n' >> src/s7.h
bin/gf format --check; echo "exit=$?"   # 期望 exit=0（s7.h 不在检查范围）
git checkout src/s7.h                    # 还原

# 配置缺失时报错退出（而非误扫默认目录）
mv gfformat.json /tmp/
bin/gf format --check; echo "exit=$?"   # 期望 error: 找不到 gfformat.json，exit=1
mv /tmp/gfformat.json .
```

## 5. 2026/06/24 gf format 统一格式化命令 + Debian 格式化检查 CI

### 5.1 What
1. 新增 `gf format` 子命令（对标 mogan 的 `tools/format/liii/goldformat.scm`），一次性统一格式化仓库内所有 C++（clang-format）与 Scheme（gf fmt）文件；`--check` 为非破坏检查模式（不改文件，发现未格式化文件则退出码 1，供 CI）。
2. 扫描范围由项目根的 `gfformat.json` 决定（`cpp-roots` / `scm-dirs`），**非硬编码**——使该命令可跨项目复用。
3. 排除规则由项目根的 `gfexclude.json` 决定，C++/Scheme 收集都尊重它；支持精确路径、文件夹（目录名）、通配符（`*`）三种 pattern。
4. 在 `gfproject.json` 注册 `format` 子命令。
5. 新增 Debian 13 格式化检查 CI（`.github/workflows/ci-format-check-debian13.yml`）：构建 `bin/gf` 后运行 `gf format --check`。

### 5.2 Why
goldfish 已有 `gf fmt`（Scheme）和 `.clang-format`（C++），但缺**统一入口**和 **CI 级格式化门禁**：C++ 此前无自动化格式化/检查，Scheme 只能逐目录手动跑。mogan 已验证过这套思路，本任务把它补进 goldfish 自身。

扫描范围配置化而非硬编码，是为了让**同一个 `gf format` 在不同仓库复用**：mogan 之前不得不维护私有 `tools/format/`（硬编码 `TeXmacs/progs/...`），正是因为 goldfish 没有该命令且无法声明范围。配置化后，mogan 等下游仓库只需放一份自己的 `gfformat.json`，即可直接用 goldfish 的 `gf format`。

排除上需要特殊对待两类文件：
- **s7 体系文件**（`src/s7*`，38 个）：clang-format 会破坏其脆弱的宏定义与排版（实测导致 `use_write_t`/`shared_info_t`/`sigjmp_buf` 等编译错误），必须整体排除。
- **测试文件**（`tests`）：含大量故意畸形的夹具（缺括号等，交给 fmt 会抛 read-error），且改动频繁，纳入仓库级 format 既慢又易误报；测试格式化由各测试自身负责。

### 5.3 How
- **三文件结构**（与 mogan 对齐）：`goldformat.scm`（主逻辑）、`goldformat-binary.scm`（定位 clang-format）、`goldformat-path.scm`（配置读取 + 文件收集）。
- **C++ 检查**：逐文件 `clang-format --dry-run --Werror <file>`，靠退出码判定（stderr/diff 重定向 `/dev/null`）；格式化模式用 `clang-format -i --files=<list>` 批量处理。
- **Scheme 检查**：逐文件起 `gf fmt --dry-run <file>` 子进程，捕获 stdout 与磁盘内容逐字节比对（与 C++ 逐文件 clang-format 对称）。之所以逐文件而非整目录：goldfish 的 `gf fmt --dry-run` **不支持目录路径**（`goldfmt.scm` 注释明确），但单文件 dry-run 只 `(display formatted)` 输出纯净结果、可安全逐字节比对。**不走同进程调 fmt 引擎**——`format-string` 走 `scan-string`（直接 read）会丢注释/空行，导致含注释文件被误判。check 慢只在 CI，本地用 `gf format`（格式化模式）即可。
- **Scheme 格式化**：对每个 scm-dir 直接调 `gf fmt <dir>`（每目录一次进程，高效），靠 gf fmt 自身的文件夹排除跳过 tests。
- **gfexclude.json 的三类 pattern**：
  - 精确路径：entry 等于 pattern、或以 `/pattern` 结尾（`tests` 匹配 `tools/fmt/tests`、`tests`）。
  - 文件夹：上述精确规则对子目录也判——`format-directory`（goldfmt.scm）和 `collect-*`（goldformat-path.scm）递归进子目录前先判排除，命中则整棵跳过。
  - 通配符 `*`：`goldformat-path.scm` 自带 `glob-match?`（只支持 `*`）。pattern 含 `/` 时对完整路径 glob（`src/s7*` 精确到 src 下），不含 `/` 时对 basename glob（`test*` 匹配任意目录下 test 开头）。
- **通配只在 goldformat-path（C++/Scheme 收集）支持，goldfmt.scm 不做通配**：`(liii path)` 的 `glob-match?` 未导出，goldfmt 用通配会 unbound；而 s7 文件只经 C++ 收集（不调 gf fmt），goldfmt 不需要通配。故 `src/s7*` 只对 C++ 生效，`tests`（精确）对 C++/Scheme 都生效。
- **配置查找**：`gfformat.json` 与 `gfexclude.json` 都放项目根（`g_project-root` 定位）；`gfformat.json` 找不到则报错退出（强制显式声明范围），`gfexclude.json` 找不到则静默降级为无排除。
- **复用既有 API**：`(os-call cmd)`（返回退出码）、`(executable)`（当前 gf 二进制）、`(liii json)` 的 `string->json`/`json-ref`、path/string 系列、`(exit N)`、`g_project-root`。
- **CI 与 mogan 的差异**：mogan 下载发布的 goldfish deb 获得 `gf`；goldfish 是自身仓库，直接 `xmake build` 后用 `bin/gf format --check`。Windows 因无 `sh` 会自动跳过 check，故只建 Debian CI。
